# CONTRIBUTING.md

> Это **открытый первоисточник** проекта (лицензия [MIT](LICENSE)). Все изменения скиллов,
> ролей, обвязки и экспериментов вносятся **сюда**, через PR. Секреты в гит не коммитить.

Правила внесения изменений на двух уровнях:

- **Сервис** — код, тесты, контракты конкретного сервиса. Разделы «Быстрый старт» … «Правила code review» ниже.
- **Стандарт / обвязка** — этот репозиторий (`AGENTS.md`, `skills/`, `docs/`, манифесты ролей). Версионирование и governance — раздел [«Версионирование и governance (пилот)»](#версионирование-и-governance-пилот).

---

## Быстрый старт

1. Клонируй репозиторий
2. Настрой окружение — см. `.env.example`
3. Запусти сервис — см. `README.md → Запуск`
4. Убедись что компонентные тесты проходят — см. `component-tests/`

## Флоу разработки

Правила разработки, коммитов и веток — в `AGENTS.md`.

Коротко:
- Ветка от `main`, живёт не дольше 1–2 дней (AGENTS.md §11)
- Коммиты по Conventional Commits (AGENTS.md §12)
- Сначала тест, потом код (AGENTS.md §5)
- Изменения API — только через спеку в `api-specification/` (AGENTS.md §2)

### Именование веток

| Префикс | Когда |
|---------|-------|
| `feat/` | новая функциональность |
| `fix/` | исправление бага |
| `test/` | только тесты |
| `docs/` | только документация |
| `chore/` | конфигурация, зависимости |

## Перед отправкой PR

- [ ] Компонентные тесты проходят (`component-tests/`)
- [ ] Юнит-тесты проходят
- [ ] `api-specification/` обновлена если изменился API
- [ ] `README.md` обновлён если изменился API, стек или способ запуска
- [ ] Коммиты по Conventional Commits
- [ ] Ветка актуальна относительно `main`

## Правила code review

**Автор PR:**
- Описание PR объясняет «зачем», не «что» (что видно из diff)
- Один PR — одно логическое изменение
- Не мержи без апрува

**Ревьюер:**
- Проверяй корректность логики, не стиль (стиль — линтер)
- Если нашёл проблему — объясни почему, не только что исправить
- Апрув = готов взять ответственность за этот код

## Документация

## Версионирование и governance (пилот)

Раздел касается изменений **стандарта / обвязки** (этот репозиторий), не кода сервиса.

### Режим пилота

- Работаем по **Trunk-Based Development** (`AGENTS.md §11`, скилл `git-conventions`): ветка от `trunk` живёт ≤ 2 дней, один PR = одно изменение, мерж только с апрувом.
- Изменения стандарта вносятся **как код** — обычным PR с ревью. Тяжёлый RFC-процесс на пилоте **не вводим**: для текущего масштаба избыточен.
- Стандарт в целом на пилоте — **pre-1.0** (тег `pilot`). Версия `1.0` фиксируется при переезде в общий инженерный handbook (см. ниже).

### Версионирование обвязки (SemVer на скилл)

Каждый скилл и манифест роли несёт `version` в шапке. Изменение = PR с ревью + **bump версии** + запись в changelog (дата, что и почему). Без версии скилл в рантайм не допускается (`docs/02_MEASUREMENT.md §5`). Это закрывает пакет **P-3** из `docs/03_REPORTING.md`.

| Bump | Когда |
|---|---|
| **MAJOR** | несовместимое изменение контракта роли или шага процесса (ломает существующий вход/выход) |
| **MINOR** | новый проход/правило/скилл, обратносовместимо |
| **PATCH** | уточнение формулировки, опечатка — без смены поведения |

Анти-дрейф: рантайм-конфигурация (модель, `skill@version`, ревизия `AGENTS.md`) фиксируется в `decisions.log`; расхождение = инцидент дисциплины (`docs/03_REPORTING.md` C-1).

### Зрелость → handbook + RFC

Когда практика **принята в стандарт** (вышла из пилота):

1. Переезжает в общий инженерный handbook — единый свод инженерных стандартов.
2. Дальнейшие изменения идут **по RFC**: предложение → обсуждение → принятие → версия.

Пилот — облегчённый режим; RFC и handbook включаются **после** пилота.
