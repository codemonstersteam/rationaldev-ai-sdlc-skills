# Skill: Code Style

**Версия:** 1.0

**Применяется на:** IMPLEMENTATION (Implementer), CODE REVIEW.
**Цель:** агент пишет код так, как принято в организации — единообразно, сопровождаемо, с учётом обязательных практик (фича-тогглы, идемпотентность, наблюдаемость).

> Это шаблон. Подставь реальные конвенции вашей организации в помеченные `<...>` места.

## Общие принципы

- **Читаемость важнее краткости.** Код пишется один раз, читается много раз.
- **Соответствие существующему коду модуля.** Перед написанием прочитай соседние файлы и следуй их стилю, даже если личное предпочтение иное.
- **Маленькие функции с одной ответственностью.**
- **Явное лучше неявного.** Никакой магии, скрытых сайд-эффектов, неочевидных глобальных состояний.

## Конвенции (заполнить под организацию)

- **Языки и версии:** `<например: Java 21 / Kotlin, Go 1.22, TypeScript 5.x>`
- **Форматирование:** автоформат обязателен (`<spotless / gofmt / prettier>`), PR без форматирования не проходит.
- **Именование:** `<camelCase для переменных, PascalCase для типов, ...>`
- **Линтеры:** `<checkstyle / golangci-lint / eslint>` — конфиг в репозитории, не отключать правила локально.
- **Структура проекта:** `<гексагональная / слоистая — описать>`

## Обязательные практики организации

### Фича-тогглы
**Весь новый функционал закрывается фича-тогглом по умолчанию.** Фича выходит в продуктив выключенной, открывается на части аудитории. 

```
if (featureToggle.isEnabled("payment-email-notifications", context)) {
    // новый код
}
```
Тоггл регистрируется в системе управления тогглами. Имя тоггла — в плане.

### Идемпотентность
Все операции изменения состояния идемпотентны. Используй идемпотентный ключ из запроса/события. Повторная обработка не создаёт дубль.

### Наблюдаемость
- **Логирование:** структурированные логи, уровни корректные, без PII/секретов (см. skill security).
- **Метрики:** ключевые операции инструментированы (счётчики, латентность).
- **Трейсинг:** прокидывай correlation/trace id через все вызовы.

### Обработка ошибок
- Не глотать исключения молча.
- Различай ретраибл и нет-ретраибл ошибки (важно для очередей).
- Сообщения об ошибках не раскрывают чувствительные данные.

### Конфигурация
- Никакого хардкода окружения, эндпоинтов, секретов. Всё из конфига/vault.

## Запрещено

- Закомментированный «мёртвый» код в PR.
- `TODO`/`FIXME` без тикета.
- Отключение правил линтера локально без обоснования.
- Изменение тестов ради прохождения CI — эскалация через fixer (Linger).
- Введение тяжёлых зависимостей без обоснования в плане.

## Артефакт на выходе

Код в PR + краткое описание PR:

```markdown
## Что сделано
Реализация ФТ-1, ФТ-2 (email-уведомления по событию payment.completed).

## Фича-тоггл
payment-email-notifications (по умолчанию OFF)

## Соответствие
- [x] Автоформат пройден
- [x] Линтер чистый
- [x] Новый функционал за тогглом
- [x] Идемпотентность по payment_id
- [x] Логи без PII
- [x] Только библиотеки из tech radar
```

## Чек-лист самопроверки

- [ ] Стиль соответствует соседнему коду модуля
- [ ] Автоформат и линтер пройдены, правила не отключены
- [ ] Новый функционал за фича-тогглом
- [ ] Операции изменения состояния идемпотентны
- [ ] Логи/метрики/трейсинг на месте, без PII
- [ ] Нет хардкода конфигурации и секретов
- [ ] Нет мёртвого кода и TODO без тикета
