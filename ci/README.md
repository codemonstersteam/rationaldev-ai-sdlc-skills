# `ci/` — тег-автоматика транка (E6). Инструмент CI/CD, НЕ агентский харнес

Ставит SemVer-тег на транк после мержа — в **той форме, что уже принята в репо** (`1.2.3` или `v1.2.3`). Запускает его **пайплайн репозитория**, не агент:
к моменту мержа харнес уже отработал и вышел. Агенты и харнес здесь не нужны — `ci/semver-bump.mjs`
зависит только от встроенных модулей Node (`node:child_process`, `node:fs`), ноль импортов из `harness/`,
ноль npm-зависимостей. Скопируйте файл в любой репозиторий — он заработает.

**Кто вызывает.** В харнесе тег ставит `@ledger` (пост-Gate #2, через `harness/close-run.mjs`) — этот же
модуль решает, какой. Пайплайн репозитория из `recipes/` — альтернатива для репо **без** харнеса;
включать оба сразу нельзя (два источника тега).

## Разделение ответственности

| Кто | Что делает | Чего НЕ делает |
|---|---|---|
| `@git-hand` (агент) | протаскивает `weight` в PR | **не бампает версию** |
| Человек | акцептует мерж на Gate #2 | не считает номер версии |
| **CI (этот инструмент)** | считает и ставит тег | не решает вес |

Источник правды версии — **сам git-тег**. Ни `VERSION`-файла, ни поля в манифесте: нечему разъезжаться.

**Форма тега — по репо.** Читаются и `1.2.3`, и `v1.2.3`; новый тег пишется той же формой, что у
**последнего релизного** тега. Тегов нет → голая форма (база `0.0.0`, первый тег бампается по весу).
Осознанный переход репо на другую форму подхватывается сам: фиксировано ПОНЯТИЕ «тег транка
авторитетен», а форма — репо-специфична.

## Контракт входа

**Вес** — из **заголовка PR** по [Conventional Commits](https://www.conventionalcommits.org/).
Единственный источник: одно место, расходиться нечему, читаемо и человеком, и машиной.

| weight | заголовок PR |
|---|---|
| patch | `fix: …` |
| minor | `feat: …` |
| major | `fix!: …` / `feat!: …`, либо футер `BREAKING CHANGE:` в теле |

**`greenfield`** передаётся только явным `--weight greenfield` (заголовком PR не выражается): первый
релиз вертикали **объявляется** `1.0.0`, а не бампается от базы.

**`!` побеждает тип:** `fix!:` — это **major**, не patch. Иначе багфикс, ломающий совместимость,
уехал бы в patch (см. `CONCEPT.md` §Классификатор — решает совместимость, а не «это баг»).

Типы без веса (`chore`, `docs`, `refactor`, `test`, …) → тега нет.

**Заголовок — ЗАПИСЬ веса, а не суждение.** `@git-hand` выводит префикс из `.agent/planner/mode`
(вес посчитал `@wirth-triage`) и ничего не выбирает сам.

Читается заголовок **PR**, а не subject мержа: при squash форж берёт заголовок PR, при merge-commit
subject выглядит как «Merge pull request #N …» и веса не несёт.

**Правило no-bump.** Если diff не трогает продуктовый код и контракт (только `.github/`, `*.md`, `docs/`,
`Dockerfile`, `Makefile`, лок-файлы, конфиги линтеров) — тега нет и канарейки нет. Плумбинг не двигает
версию API. Списки — `PLUMBING` в `semver-bump.mjs`; правьте под свой репозиторий.

Денилист **fail-closed**: путь считается плумбингом, только если заведомо им является; всё неизвестное —
продуктовое. Лишний патч-тег дёшев, потерянная версия продуктового изменения — нет. По той же причине
смешанный diff (`README.md` + `src/order.go`) **бампает**.

## Использование

```bash
# из git напрямую (теги и diff HEAD~1..HEAD берёт сам)
node ci/semver-bump.mjs --weight patch --from-git

# из подготовленных файлов (так делает CI — см. рецепты)
node ci/semver-bump.mjs --pr-title title.txt --pr-body body.txt \
                        --files-from files.txt --tags-from tags.txt --json
```

Вывод: `1.4.3<TAB>1.4.2 + patch` либо `NO_TAG<TAB><причина>`. Код возврата **всегда 0** — «тег не нужен»
это штатный исход, а не сбой; CI различает по пустому `tag`, а не по exit code.

## Рецепты

- **GitHub Actions** — `recipes/github-actions.yml` → положить в целевой репозиторий как
  `.github/workflows/tag-on-merge.yml`. Тригер `pull_request: closed` + `if: merged == true`: PR-контекст
  доступен напрямую, не надо искать PR по merge-SHA. Требует `permissions: contents: write`.
- **GitLab CI** — `recipes/gitlab-ci.yml`.
- **Любой другой** — вызовите скрипт, положите тег. Три строки:

```bash
TAG=$(node ci/semver-bump.mjs --weight "$WEIGHT" --from-git | cut -f1)
[ "$TAG" = "NO_TAG" ] || git tag -a "$TAG" -m "$TAG" && git push origin "$TAG"
```

## Список файлов diff'а — берите у API, а не из git

В рецептах намеренно `gh pr view --json files`, а не `git diff HEAD~1 HEAD`. При **squash**-мерже второе
даёт верный ответ, при **merge-commit** — нет (`HEAD~1` это второй родитель). Стратегия мержа не должна
влиять на решение no-bump. Флаг `--from-git` удобен локально, но в CI предпочитайте список от форжа.

## Канарейка

`canaryTag()` даёт pre-release `X.Y.Z-canary.N` до промоушена. Pre-release-теги **не двигают базу версии**:
`latestRelease()` их игнорирует, поэтому канарейки не сбивают счёт. Промоушен канарейки в релиз — шаг
деплой-пайплайна (Gate #3), здесь не автоматизирован.
