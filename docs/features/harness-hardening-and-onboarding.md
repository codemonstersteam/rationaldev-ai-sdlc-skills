# План: починка блокеров, установка, онбординг чужого репо

> Статус: **принят оператором, не начат**. Дата фиксации: 22.07.2026, ветка `docs/harness-hardening-plan`.
> Документ существует, чтобы работу можно было продолжить после перерыва: он содержит решения оператора,
> найденные дефекты с точными местами и очерёдность. Источники правды остаются прежними —
> `requirements/semver-verticals.md` (FRD), `docs/flows/*`, `harness/agents/_shared/*`.

## Зачем

Планировался живой прогон вертикали `patch` на `pinout/pinout-asyncapi`. Три независимые проверки
(консистентность плана · сквозной проход по пяти флоу · аудит установки) нашли, что: (а) целевой репозиторий
не по стандарту и штатно не принимается, (б) в самом харнесе есть пять блокеров, из-за которых прогон
измерял бы дефекты стенда, а не вертикаль. Отсюда очерёдность: сначала харнес, потом стенд, потом прогон.

## Решения оператора (приняты, не пересматривать без него)

| Вопрос | Решение |
|---|---|
| Каталог канонического клона | остаётся **`~/.rationaldev`** (без дефиса) |
| Базовая версия при онбординге репо без тегов | **оставить как есть** — первый патч даёт `v0.0.1` |
| `validate-layout` на репо без `internal/<slug>/` | не молчать: печатать «инвариант не действует» **и выпускать тикет-долг** на рефакторинг раскладки |
| Модели для claude/codex | **override для всех раннеров**, клон остаётся read-only |
| Тоггл в greenfield | **не нужен** — первый релиз никого не ломает, транк релизится один раз тегом `1.0.0` |
| `--require-additive` | **поведение по умолчанию**; отдельного включателя не будет |

## P9 · Починка блокеров (первым делом, до любого прогона)

1. **`contract-diff` — блокировка по умолчанию.** Убрать advisory-ветку и флаг-включатель: breaking → exit 2
   всегда. Единственное исключение — вес `major`, и оно выводится из `.agent/planner/mode`, а не из аргумента
   командной строки (поканйока: флага нет — забыть или злоупотребить нечем).
   **Дыра, которую дефолт не закрывает:** `harness/validate-contract-diff.mjs:94` структурно диффит только
   `.json`; типовой контракт профиля `service` — `api-specification/openapi.yaml`, то есть YAML → «неоценимо»
   → exit 0. При блокирующем дефолте честная позиция — **fail-closed** (не доказали → STOP). Чтобы это не
   заперло все сервисы: `@wirth-apidesigner` при заморозке кладёт рядом машинную копию `openapi.json`, дифф
   идёт по ней — ноль внешних зависимостей, плюс стабильный артефакт для диффа.
2. **Вайп прогона неполный** (`harness/close-run.mjs:47`): не чистятся `.agent/plan-reviewer/` (вердикт
   `plan-review.md` и счётчик раундов) и `.agent/planner/{frd.md,slices.md,target,network-topology.md,
   rollout-plan.md}`, `.agent/release-health/`. Протухший вердикт удовлетворяет жёсткий блок имплементера на
   **следующей** задаче — `@hughes` проходит без реального ревью нового плана.
3. **`planReadyForApproval` не привязан к текущей работе** (`harness/enforcement/shared.mjs:74`): «план собран»,
   если хоть где-то в `docs/design/*/` есть `PLAN.md`. После первого greenfield защита от преждевременного
   `GATE1 APPROVE` мертва навсегда. Привязать к `.agent/planner/change-dir` / `chore-dir` / текущему срезу.
4. **`level=trivial` недрайвится.** `harness/agents/_shared/izi.md:167` шлёт задачу прямо к `@hughes`, минуя
   Gate #1 и рабочую ветку, а гардрейл (`harness/enforcement/claude/gate-check.mjs:48-95`) блокирует любого
   имплементера без `gate1.approved` + `plan-review.md` и на транке — удовлетворить его нечем. Решение выбрать
   при реализации: мини-Gate #1 по образцу chore **или** отдельный токен режима, который знает гардрейл.
5. **`@wirth-moduledesigner` не знает про `change-dir`** (`harness/agents/_shared/wirth-moduledesigner.md:57-65`):
   сентинел идемпотентности смотрит на greenfield-путь `docs/design/<slice>/module-tree.md`, уже помеченный
   `DONE:`. На `minor`/`major` (там `design=needed` по умолчанию) роль вернёт «idempotent», ничего не написав,
   и `@wirth-ticketer` останется без входа. Завести SemVer-режим с сентинелом на `<change-dir>/module-tree.md`.
6. **Раскладка — тикет-долг вместо тихого зелёного.** `harness/lib/validators.mjs:430` матчит только
   `internal/<slug>/`; на репо без такого каталога путей ноль → «нечего валидировать», exit 0, гейт выглядит
   рабочим. Валидатор обязан явно печатать, что инвариант не действует, а прогон — выпускать **тикет-долг** на
   рефакторинг раскладки в append-only `docs/changes/DEBT.md`; оператор заводит из него отдельный прогон.
7. **Мелочи того же захода:**
   - `TAG_EXISTS`: `harness/close-run.mjs:121` молча продолжает, хотя `docs/flows/patch-flow.md:51` обещает отказ;
   - chore: поканйока «зелёный без артефакта» отключена структурно — `harness/enforcement/claude/gate-bash.mjs:19`
     ищет тикет, которого у chore нет;
   - стейл-счётчик раундов `@mills` уводит первый же блокер нового прогона в эскалацию (следствие п.2);
   - **археология релиза:** номер PR в аннотации тега уже есть (`v0.0.1 — v0.0.0 + patch (PR #1)`), но обратной
     связи нет. Добавить best-effort после успешного тега: GitHub Release от тега и комментарий в PR
     «released as `<tag>`». Не легли (прав нет) — прогон не рушится, строка результата это называет.

**Приёмка P9:** все проверки харнеса зелёные (`node --test harness/test/*.mjs`, `claude-hooks.smoke`,
`harness/smoke/run.sh`), плюс новые тесты на каждый пункт 1–6.

## P4 · Установка и обновление

- Override моделей `~/.config/rationaldev/models.<runner>.json` **для всех раннеров**. Сейчас
  `install.sh:39-43` подставляет `RATIONALDEV_MODELS` только для `opencode`; для claude/codex
  `harness/configure-models.mjs:37` пишет прямо в клон → вместе с перегенерацией проекций это **38 изменённых
  файлов**, и `rationaldev update` отказывается (pristine-guard, честно, но блокирует).
- То же в `install.ps1` — там механизма override **нет вовсе**, ни для одного раннера.
- Парадокс, который надо снять: `rationaldev install <path>` жёстко ставит `--no-input`, поэтому клон не
  пачкает, но и модели для claude/codex через него настроить **нельзя** — только дефолт.
- `.claude/settings.json` в проекте — реальный файл и при повторной установке **не обновляется** (кладётся
  `settings.harness.json` рядом с «слей вручную»), поэтому новая проводка хуков до репо сама не доедет.
  Сделать слияние проводки.
- Документация: абзац про override в `harness/README.md` (сейчас только в комментариях кода).
- Смоук: «интерактивная установка не пачкает клон» (сейчас `harness/smoke` проверяет синтетический `dirt.txt`,
  а не реальный путь через `configure-models.mjs` — из-за чего регрессия и осталась незамеченной).

## P0 · Амендмент FRD

`requirements/semver-verticals.md:185` (NFR-1) утверждает «только native-репозитории, разведка удалена», и это
продублировано во всех пяти флоу. Новый маршрут онбординга обязан быть записан как **осознанная ревизия**:
онбординг — **одноразовая натурализация** репозитория (навязываем свой формат и получаем native-репо), а не
возврат conform-режима (подстройки под чужую парадигму). Без этого пункта FRD противоречит коду.

## P1–P3 · Онбординг чужого репозитория

- Роль **`@wirth-onboard`** с двумя режимами: `onboard` (пакета нет — восстановить из кода
  `docs/design/slice-*/{module-tree,contracts,c4}.md`, `CONTEXT.md`, `api-specification/`, `AGENTS.md`) и
  `reconcile` (пакет есть, но отстал от кода или от версии стандарта). Пишет только `docs/` и
  `api-specification/`, продуктовый код не трогает.
- **Провенанс обязателен:** каждая запись восстановленного пакета несёт пометку `as-is` (так устроено) или
  `gap` (расхождение с документацией) — иначе онбординг узаконит кривизну задним числом. Проверяется
  валидатором, а не глазом `@mills`.
- Маршрут `route=onboard`: no-bump **уже бесплатен** (`ci/semver-bump.mjs:91` возвращает `tag=null` для любого
  нераспознанного веса) — нужны только легальный токен маркера (`wirth-triage.md`, белый список),
  строка в таблице роутинга (`izi.md`), запись в `PIPELINE` (`enforcement/shared.mjs`), `ORDER`
  (`gen-agents.mjs`) и подтверждение, что `@ledger` на этом маршруте тоже закрывает прогон.
- **Шестой флоу-док** `docs/flows/onboard-flow.md` (ревизия пяти существующих — отдельная работа).
- **Валидатор полноты пакета** `harness/validate-package.mjs` — узко: наличие обязательных артефактов +
  размеченный провенанс. Не подменять `validate-contract-frozen`/`validate-dod`, которые на чужой контракт
  не ложатся без отдельной работы над `harness/target-profiles.json` (профиль `cli` ждёт
  `config.schema.json`+`report.schema.json`).
- **`.agent/planner/target` не пишет никто** на SemVer-вертикалях → `harness/validate-dod.mjs:19` берёт профиль
  `service` и ищет `openapi.yaml` в CLI-проекте, падая в самом конце цикла. Онбординг обязан его писать.
- Приёмка пакета — **под-шаг Gate #1**, а не четвёртый гейт (решение «три гейта» не пересматривается).

## P7–P8 · Тоггл и актуализация документации

- **Тоггл по весам:** `greenfield` — не нужен вовсе (ломать нечего, релиз один — `1.0.0`; правка ячейки
  `CONCEPT.md:92`); `minor` — способность default OFF, уже прописано полностью; `major` — переключатель
  совместимости / миграционное окно фиксируется в ADR и проверяется на приёмке; `patch` — вернуть оговорку про
  рискованный фикс за тогглом как опцию оператора.
- **Документация после работы.** Сейчас автор актуализации есть только в greenfield (`@dijkstra` + `validate-readme`);
  на `patch/minor/major` `@dijkstra` не запускается (`izi.md:234`), а `@wirth-ticketer` запрещает README-тикет
  (`wirth-ticketer.md:77,116-118`). Итог — тупик: `@fagan` вправе забраковать устаревший README, а произвести
  правку некому. Назначить автора (вызов `@dijkstra` в конце `minor`/`major` **или** doc-тикет от тикетера,
  когда дельта трогает документированное поведение).
- **Сведение дизайн-пакета:** после мерджа изменения из `<change-dir>` должны попасть в `module-tree/contracts/c4`
  среза, иначе накапливается ровно тот дрейф, ради которого заводится `reconcile`.

## P5 · Эмуляция прогона (репетиция за оператора)

Пройти весь путь руками на `/tmp`-копии `pinout-asyncapi` с одноразовым приватным репозиторием на форже:
онбординг → патч → PR → мерж → тег → `LEDGER`. Отдать оператору чек-лист граблей. Контур RUN-CLOSE проверяется
именно так — локальный bare-remote не годится (`harness/close-run.mjs` читает PR через `gh`).

**Сценарий патча уже найден и доказан на данных:** `contract-tests.yaml:22-28` и `docs/integration-netlist.md:9`
документируют `save_json_report` / `json_report_file` как работающее поведение, но код их не использует
(`validator/types.go:39-43` объявлены, применяется только `Timeout`) — файл отчёта не создаётся. Починка
обратно совместима: запись включается только при `save_json_report: true`, а во всех существующих конфигах
секции `settings` нет. Запасной сценарий `minor` — флаг `--all-channels` (сейчас
`validator/contract_validator.go:67-72` берёт первый канал и молча игнорирует остальные).
