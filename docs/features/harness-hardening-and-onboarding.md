# План: починка блокеров, установка, онбординг чужого репо

> Статус: **принят оператором, не начат**. Дата фиксации: 22.07.2026, ветка `docs/harness-hardening-plan`.
> Документ существует, чтобы работу можно было продолжить после перерыва: он содержит решения оператора,
> найденные дефекты с точными местами и очерёдность. Источники правды остаются прежними —
> `requirements/semver-verticals.md` (FRD), `docs/flows/*`, `harness/agents/_shared/*`.

## Зачем

Планировался живой прогон вертикали `patch` на `pinout/pinout-asyncapi`. Три независимые проверки
(консистентность плана · сквозной проход по пяти флоу · аудит установки) нашли, что: (а) целевой репозиторий
не по стандарту и штатно не принимается, (б) в самом харнесе есть пять блокеров, из-за которых прогон
измерял бы дефекты стенда, а не вертикаль. Отсюда очерёдность: сначала харнес, потом стенд, потом прогон.

## Решения оператора (приняты, не пересматривать без него)

| Вопрос | Решение |
|---|---|
| Каталог канонического клона | остаётся **`~/.rationaldev`** (без дефиса) |
| Базовая версия при онбординге репо без тегов | **оставить как есть** — первый патч даёт `v0.0.1` |
| `validate-layout` на репо без `internal/<slug>/` | не молчать: печатать «инвариант не действует» **и выпускать тикет-долг** на рефакторинг раскладки |
| Модели для claude/codex | **override для всех раннеров**, клон остаётся read-only |
| Тоггл в greenfield | **не нужен** — первый релиз никого не ломает, транк релизится один раз тегом `1.0.0` |
| `--require-additive` | **поведение по умолчанию**; отдельного включателя не будет |

## P9 · Починка блокеров (первым делом, до любого прогона)

1. **`contract-diff` — одно поведение и отраслевые инструменты.** Решение оператора: харнес **всегда** работает
   как `--require-additive`, другого поведения нет. Значит у инструмента остаётся **одна** реакция — нашёл слом,
   печатает список, выходит с кодом 2. Ни флагов, ни чтения `.agent/planner/mode`, ни ветки под `major`:
   инструмент не знает про веса и тестируется чистыми юнит-тестами.
   **Разница по весам живёт в конвейере, а не в инструменте:** `minor` — код 2 это STOP (вес определён неверно
   → ре-триаж как `major`); `major` — код 2 ожидаем, напечатанный список **и есть** артефакт (в тело PR как
   `BREAKING CHANGE` и во вход миграционного тикета), прогон не останавливается, потому что слом уже принят
   весом на Gate #1; `patch` — код 2 значит, что патч тронул контракт, чего он не вправе → STOP.
   **Дыра, которую это не закрывает:** `harness/validate-contract-diff.mjs:94` структурно диффит только `.json`
   (Node читает JSON из коробки, YAML — нет, а зависимостей харнес не тянет); типовой контракт профиля `service`
   — `api-specification/openapi.yaml` → «неоценимо» → exit 0. На сервисах гейт не проверяет **ничего**.
   **Решение оператора — брать отраслевые дифферы**, а не дописывать свой: валидатор становится тонким
   **диспетчером** по формату спеки — OpenAPI → `oasdiff breaking`, AsyncAPI → `asyncapi diff`, JSON Schema →
   встроенный структурный дифф (он уже есть), инструмента нет → **STOP** с точной командой установки
   (fail-closed). Самописная копия `openapi.json` не нужна — отменяется.
   **Два рубежа вместо одного (ранняя проверка важнее блокирующей):**
   (а) сразу после эволюции контракта, до нарезки тикетов — слом там стоит одного ре-триажа, тот же слом на
   приёмке стоит всей реализации; (б) в CI целевого репо как проверка на PR — тогда слом видит и человек, и он
   ловится в правках, сделанных руками мимо харнеса (рецепт в `ci/recipes/`, как для тег-автоматики).
   **Цена, названная вслух:** появляется внешняя зависимость — харнес перестаёт быть самодостаточным на
   репозиториях со спекой. Взамен — настоящий отраслевой классификатор ломающих изменений вместо пяти самописных
   правил, которые не знают ни про `enum`, ни про сужение диапазонов, ни про параметры запроса.
2. **Вайп прогона — инверсия (решение оператора).** Сейчас `WIPE` (`harness/close-run.mjs:47`) — это список
   того, что стирать, и он отстаёт: не чистятся `.agent/plan-reviewer/` (вердикт `plan-review.md` + счётчик
   раундов), `.agent/planner/{frd.md,slices.md,target,network-topology.md,rollout-plan.md}`,
   `.agent/release-health/`. Протухший `plan-review.md` от закрытой задачи удовлетворяет жёсткий блок
   имплементера на **следующей** — `@hughes` проходит без реального ревью нового плана; стейл `round≥1` уводит
   первый блокер нового прогона в лишнюю эскалацию.
   **Починка — обратный принцип:** стирать `.agent/` **целиком**, перечисляя лишь то, что нужно **сохранить**
   (белый список). Любой новый артефакт по умолчанию вычищается — забыть его в списке больше нельзя, закрывается
   весь класс, а не один пропуск. Кандидат в сохраняемое ровно один — `.agent/memory.md` (долговременная память,
   скилл `memory`); при реализации проверить, действительно ли он переживает задачу — если нет, белый список
   пуст и вайп = «снести `.agent/` и создать заново». Тест на полноту обязателен: `harness/test/close-run.test.mjs:40`
   сейчас `plan-reviewer/` не покрывает — из-за чего регрессия и осталась незамеченной.
3. **`planReadyForApproval` не привязан к текущей работе** (`harness/enforcement/shared.mjs:74`): «план собран»,
   если хоть где-то в `docs/design/*/` есть `PLAN.md`. Но `PLAN.md` — **вечный** артефакт дизайн-пакета (часть
   Doc-as-Code, коммитится в git, живёт после задачи); значит после первого greenfield глобальный вопрос «есть
   ли хоть один план» всегда истинен → защита от преждевременного `GATE1 APPROVE` мертва навсегда.
   **Починка:** спрашивать про план **текущей** задачи по адресу из состояния прогона —
   `.agent/planner/change-dir` (SemVer/онбординг) · `chore-dir` (chore) · текущий срез из `slices.md`
   (greenfield), — а не про «любой durable `PLAN.md`». Вечную запись проверяем через временный указатель на неё,
   а не через «существует где-нибудь вообще». **Заодно** пройти `shared.mjs` на другие проверки той же болезни
   (глоб durable-артефакта — тикеты, контракт — вместо привязки к текущей работе): корень один, чинить в одном заходе.
4. **`level=trivial` удалить (решение оператора).** `izi.md:167` шлёт trivial прямо к `@hughes`, минуя Gate #1
   и рабочую ветку, а гардрейл (`gate-check.mjs:48-95`) блокирует любого имплементера без `gate1.approved` +
   `plan-review.md` и на транке — удовлетворить нечем; сегодня «работает» только на протухшем `.agent/` от
   прошлой задачи (после починки п.2–3 встанет окончательно). Ветка досталась от досемверной эпохи и дублирует
   веса: правка существующего кода в один модуль без смены контракта — это **`patch`** (там уже есть
   `design=skip`); новый код в один модуль — вырожденный **greenfield-`modular`**. Убрать единственную ветвь,
   обходящую Gate #1: вычистить `trivial` из таблицы роутинга и обхода `@hughes` (`izi.md`), из Axis 2 и
   возвратной строки (`wirth-triage.md`), из инварианта веса (`greenfield-flow.md`); грепом убедиться, что
   `trivial` не зашит в валидаторах/enforcement/тестах (не оставить осиротевшую ссылку).
5. **`@wirth-moduledesigner` не знает про `change-dir`** (`harness/agents/_shared/wirth-moduledesigner.md:57-65`):
   сентинел идемпотентности (маркер `DONE:`, защита от повторной работы) смотрит на greenfield-путь
   `docs/design/<slice>/module-tree.md`, который помечен `DONE:` с первой сборки среза. Но на SemVer-вертикали
   роль обязана писать в `<change-dir>/`. Итог: на `minor`/`major` (там `design=needed` по умолчанию) роль в
   первую секунду видит чужой `DONE:`, возвращает «idempotent» и **ничего не пишет** — дизайн-шаг молча
   пропущен, `@wirth-ticketer` без входа. Механический недосмотр перевода на вертикали: родственные
   `@wirth-apidesigner`/`@wirth-planner` уже читают `change-dir`, эту забыли. **Починка:** SemVer-режим — роль
   читает `.agent/planner/change-dir`, выход **и сентинел** смотрят на `<change-dir>/module-tree.md`. При
   реализации проверить, не завязан ли сентинел тем же greenfield-путём и на `contracts.md`/`c4.md` — тогда
   чинить все три выхода, а не один.
6. **Раскладка — тикет-долг в `/debt/` вместо тихого зелёного (решение оператора, фаза 1).**
   `harness/lib/validators.mjs:430` матчит только `internal/<slug>/`; на репо без такого каталога путей ноль →
   «нечего валидировать», exit 0 — ложное «всё в порядке». Валидатор обязан явно печатать «инвариант раскладки
   не действует — репозиторий не по стандарту», а прогон — выпускать **тикет-долг** `/debt/task-NNN.md` с чёткой
   постановкой (что не так · что сделать · критерий готовности). Долг живёт **рядом с кодом**, а не во внешнем
   трекере — «доказательство преступления» видно на ревью и в той же ветке.
   **Инвариант: `/debt/` пустой = чисто.** Это открытая очередь; оператор заводит из файла отдельный прогон,
   который приводит раскладку к стандарту. Когда долг закрыт — `close-run` **удаляет** `/debt/task-NNN.md`
   (исчезновение файла = доказательство оплаты), а вечная запись «что было должно, когда закрыто» остаётся в
   `docs/changes/LEDGER.md`. Та же двухслойность, что и везде: `/debt/` — состояние-очередь, `LEDGER` — история.
   **Нумерация — локальный сквозной `NNN`**, никакого внешнего трекера: долг целиком в репозитории.
   Прогон при этом не заложник чужой раскладки — патч продолжается, долг записан.
7. **Мелочи того же захода:**
   - `TAG_EXISTS`: `harness/close-run.mjs:121` молча продолжает, хотя `docs/flows/patch-flow.md:51` обещает отказ;
   - chore: поканйока «зелёный без артефакта» отключена структурно — `harness/enforcement/claude/gate-bash.mjs:19`
     ищет тикет, которого у chore нет;
   - стейл-счётчик раундов `@mills` уводит первый же блокер нового прогона в эскалацию (следствие п.2);
   - **археология релиза:** номер PR в аннотации тега уже есть (`v0.0.1 — v0.0.0 + patch (PR #1)`), но обратной
     связи нет. Добавить best-effort после успешного тега: GitHub Release от тега и комментарий в PR
     «released as `<tag>`». Не легли (прав нет) — прогон не рушится, строка результата это называет.

**Приёмка P9:** все проверки харнеса зелёные (`node --test harness/test/*.mjs`, `claude-hooks.smoke`,
`harness/smoke/run.sh`), плюс новые тесты на каждый пункт 1–6.

## P0 · Амендмент FRD

`requirements/semver-verticals.md:185` (NFR-1) утверждает «только native-репозитории, разведка удалена», и это
продублировано во всех пяти флоу. Новый маршрут онбординга обязан быть записан как **осознанная ревизия**:
онбординг — **одноразовая натурализация** репозитория (навязываем свой формат и получаем native-репо), а не
возврат conform-режима (подстройки под чужую парадигму). Без этого пункта FRD противоречит коду.

## P1–P3 · Онбординг чужого репозитория

- Роль **`@wirth-onboard`** с двумя режимами: `onboard` (пакета нет — восстановить из кода
  `docs/design/slice-*/{module-tree,contracts,c4}.md`, `CONTEXT.md`, `api-specification/`, `AGENTS.md`) и
  `reconcile` (пакет есть, но отстал от кода или от версии стандарта). Пишет только `docs/` и
  `api-specification/`, продуктовый код не трогает.
- **Провенанс обязателен:** каждая запись восстановленного пакета несёт пометку `as-is` (так устроено) или
  `gap` (расхождение с документацией) — иначе онбординг узаконит кривизну задним числом. Проверяется
  валидатором, а не глазом `@mills`.
- Маршрут `route=onboard`: no-bump **уже бесплатен** (`ci/semver-bump.mjs:91` возвращает `tag=null` для любого
  нераспознанного веса) — нужны только легальный токен маркера (`wirth-triage.md`, белый список),
  строка в таблице роутинга (`izi.md`), запись в `PIPELINE` (`enforcement/shared.mjs`), `ORDER`
  (`gen-agents.mjs`) и подтверждение, что `@ledger` на этом маршруте тоже закрывает прогон.
- **Шестой флоу-док** `docs/flows/onboard-flow.md` (ревизия пяти существующих — отдельная работа).
- **Валидатор полноты пакета** `harness/validate-package.mjs` — узко: наличие обязательных артефактов +
  размеченный провенанс. Не подменять `validate-contract-frozen`/`validate-dod`, которые на чужой контракт
  не ложатся без отдельной работы над `harness/target-profiles.json` (профиль `cli` ждёт
  `config.schema.json`+`report.schema.json`).
- **`.agent/planner/target` не пишет никто** на SemVer-вертикалях → `harness/validate-dod.mjs:19` берёт профиль
  `service` и ищет `openapi.yaml` в CLI-проекте, падая в самом конце цикла. Онбординг обязан его писать.
- Приёмка пакета — **под-шаг Gate #1**, а не четвёртый гейт (решение «три гейта» не пересматривается).

  правку некому. Назначить автора (вызов `@dijkstra` в конце `minor`/`major` **или** doc-тикет от тикетера,
  когда дельта трогает документированное поведение).

## P5 · Эмуляция прогона (репетиция за оператора)

Пройти весь путь руками на `/tmp`-копии `pinout-asyncapi` с одноразовым приватным репозиторием на форже:
онбординг → патч → PR → мерж → тег → `LEDGER`. Отдать оператору чек-лист граблей. Контур RUN-CLOSE проверяется
именно так — локальный bare-remote не годится (`harness/close-run.mjs` читает PR через `gh`).

**Сценарий патча уже найден и доказан на данных:** `contract-tests.yaml:22-28` и `docs/integration-netlist.md:9`
документируют `save_json_report` / `json_report_file` как работающее поведение, но код их не использует
(`validator/types.go:39-43` объявлены, применяется только `Timeout`) — файл отчёта не создаётся. Починка
обратно совместима: запись включается только при `save_json_report: true`, а во всех существующих конфигах
секции `settings` нет. Запасной сценарий `minor` — флаг `--all-channels` (сейчас
`validator/contract_validator.go:67-72` берёт первый канал и молча игнорирует остальные).
