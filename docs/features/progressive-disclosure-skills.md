# План #40 — progressive disclosure скиллов (тела грузить по требованию)

> Тикет исправления анти-паттерна. Ветка: `feat/skill-progressive-disclosure`.
> Реализует рекомендацию №5 из [`../agent-best-practices.md`](../agent-best-practices.md).

## Контекст

> **Реком. №5:** скиллы через progressive disclosure — в контексте только `name`+`description`;
> тело грузится **при активации**; крупные `SKILL.md` дробить на файлы «по требованию».
> → Это буквально #40 и правило «SKILL ≤300 строк». **Сейчас тела встраиваются целиком —
> анти-паттерн, который отрасль уже отвергла** (Anthropic Agent Skills; OMO `tools/skill`).

Прогрессивное раскрытие = три стадии: **discovery** (в контексте только имя+описание) →
**activation** (при совпадении задачи агент подгружает тело `SKILL.md`) → **execution**
(следует инструкции, при нужде читает companion-файлы).

## Проблема и её цена (замерено)

Роль получает **полные тела** `SKILL.md` в системном промпте, и они пересылаются в **каждом**
запросе цикла субагента. Детектируется `run-summary.mjs` / `skill-embedding-cost.mjs` по
frontmatter `name:` / H1 во `flow.jsonl`.

**Baseline (прогон #4, [`../../experiments/token-bench/skill-embedding-baseline.md`](../../experiments/token-bench/skill-embedding-baseline.md)):**
- встроено скиллами: **~544.8K токенов = 10.1 %** всего input прогона;
- потолок экономии (повторные пересылки): **~521.7K (9.7 %)**;
- доминирует `component-tests` — **322K net** (большой скилл × 79 запросов в циклах scaffolder/wirth-tester).

**Корень:** скилл отдаётся роли как **текст в системном промпте**, а не как загружаемый по
требованию ресурс. Стадий discovery→activation→execution нет.

## Фаза 0 — разведка (обязательна ПЕРЕД кодом)

Определить, **как opencode 1.17 сейчас подаёт скиллы** (у нас `.opencode/skills/*` — симлинки на `skills/lib/*`):

- **(A)** opencode инъектит **полное тело** скилла в системный промпт; **или**
- **(B)** opencode уже делает disclosure (в контекст — только описание, тело по вызову), а полные
  тела во `flow.jsonl` появились потому, что **тела ролей велят «Грузи по имени скилл X»** и агент
  читает файл целиком заранее.

**Как проверить:** один `opencode run` на минимальной роли; во `flow.jsonl` (dir=out, первый запрос)
посмотреть `messages[0]` (system): есть ли там полные тела `SKILL.md` до какого-либо обращения к скиллу.
- тела в system до активации → **путь (A)** (нужен свой tool `skill`);
- тел нет, появляются только после «чтения» → **путь (B)** (чинить формулировки ролей).

Вывод фазы 0 фиксирует развилку и определяет объём фаз 1–3.

## Фаза 1 — Discovery: в контексте только name+description

- Источник реестра — [`../../skills/INDEX.json`](../../skills/INDEX.json) (`name` + `description` каждого скилла).
- В контекст роли попадает **только** этот реестр (что уже и так частично есть в манифесте роли),
  **без тел**. Полное тело — не в системном промпте.

## Фаза 2 — Activation: тело по требованию

- **Путь (A):** в rational opencode-плагин добавить tool **`skill(name)`** (модель OMO
  `packages/omo-opencode/src/tools/skill/tools.ts`): читает `skills/lib/<name>/SKILL.md`, возвращает тело;
  **сессионный кэш** (не грузить дважды за сессию). Описание тула перечисляет доступные скиллы (discovery).
- **Путь (B):** переписать тела ролей: «Грузи по имени скилл X» → **«при активации задачи вызови
  `skill('X')`»**, чтобы агент не втягивал файл заранее.

## Фаза 3 — Delegation: `load_skills` из тикета

`skills:` тикета (уже валидируется `expectedTicketSkills`, см. `harness/lib/validators.mjs`)
прокидывать в делегацию как **preload-список** (аналог OMO `task(load_skills=[...])`): субагент
активирует **ровно** набор из тикета. Смыкает **дизайн-гарантию** (валидатор до Gate #1) и
**рантайм-факт** (что реально активировано, MUST-блок `run-summary`).

## Фаза 4 — Дробление крупных SKILL.md

`component-tests` (самый дорогой, 4185 ток) и подобные — редко-используемые части вынести в
companion-файлы, грузимые по ссылке. Правило «≤300 строк» + progressive disclosure на уровне файлов
(execution-стадия читает companion лишь при нужде).

## Критерии приёмки

- Во `flow.jsonl` **нет полных тел `SKILL.md` до вызова `skill()`** (детектор `name:`/H1 из
  `skill-embedding-cost.mjs` = 0 до активации).
- tool `skill()` (если путь A) возвращает корректное тело; сессионный кэш не грузит дважды.
- **Замер before/after** тем же `experiments/token-bench/runners/skill-embedding-cost.mjs`:
  net встроенных скиллов → **~0**; падение input-токенов подтверждено (`usage.jsonl` + `run-summary.mjs`)
  **без потери качества артефактов**.
- MUST-инвариант `run-summary` (скиллы ⊆ манифеста роли) по-прежнему зелёный.

## Риски

- **Слабая модель (Qwen) может «забыть» вызвать `skill()`** → подстраховка: хук
  `category-skill-reminder` (из OMO) инжектит «нужные скиллы этого тикета: …, вызови `skill([...])`».
- **Нативный механизм opencode может конфликтовать со своим тулом** → фаза 0 снимает эту
  неопределённость до кода (если путь B — свой тул не нужен вовсе).
- Дробление `SKILL.md` рискует «спрятать» важное правило → выносить только действительно
  контекст-зависимые части, ядро оставлять в `SKILL.md`.
