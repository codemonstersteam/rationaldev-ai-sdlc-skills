
# Целевой процесс агентской разработки ПО

> Целевая модель производства ПО на базе агентов «izi». Вес работы задаёт **SemVer**; пять
> вертикалей (`greenfield · patch · minor · major · chore`) идут по общему хребту и отличаются
> головой конвейера. Пошагово по весу — [`flows/`](flows/); формальные требования —
> [`../requirements/semver-verticals.md`](../requirements/semver-verticals.md); граф делегирования —
> [`harness-flow.md`](harness-flow.md).

## 1. Контекст и границы

Команда продукта: человек + машина. Вокруг выстроена зрелая автоматизация CI/CD: каждый коммит едет
в прод, новое закрыто фиче-тогглами и раскатывается канарейкой на часть аудитории. Тестовых
сред/стендов нет — есть CI и канареечные релизы прямо в прод.

Жизненный цикл разделён на две стадии:

- **Discovery** — гипотезы, функциональные и нефункциональные требования. Ведёт
  **инженер-разработчик совместно с ИИ** (отдельной роли-аналитика нет).
- **Delivery** — реализация задачи. **Здесь работают агенты.** Граница входа — момент, когда
  требование сформулировано.

**Область применимости — строгая:** харнес ведёт **репозитории, которые построил сам** (замороженный
контракт `api-specification/`, дизайн-пакет `docs/design/slice-*/`, своя парадигма тестов). Стандарт
задаётся, а не подбирается: разведки чужих парадигм и conform-режима в конвейере нет.

## 2. Распределение моделей — два тира

| Тир | Кто | Почему |
|---|---|---|
| **large** (суждение) | `izi`-роутинг, `@gilb`, `@wirth-*`, `@mills`, `@fagan`, `@linger` | классификация, проектирование, ревью, приёмка |
| **small** (исполнение) | `@hughes`/`@hughes-rework`, `@scaffolder`, `@git-hand`, `@ledger`, `@michtom` | предсказуемая механика с высокой частотой вызовов |

Принцип: **асимметрия генератор/критик** — кто генерирует артефакт (план, код, выкат), тот его не
принимает; оценщик не мельче генератора. Это ключ к тому, чтобы цикл сходился к качеству, а не к
«зелёному любой ценой». Привязка ролей к моделям — `harness/models.config.json`.

## 4. Вес работы — классификатор SemVer 2.0.0

Вес определяет `@wirth-triage`, **дословно по SemVer 2.0.0**; решающая ось одна — **обратная
совместимость документированного контракта**. `izi` роутит по вердикту механически и **никогда не
переклассифицирует** (например, `patch` в `minor` из-за «большого диффа»).

0. Не трогает секрет модуля и контракт, не добавляет поведения, которое утверждал бы компонентный
   тест → **`chore`**.
1. Кода/публичного API ещё нет → **`greenfield`** (уровень: `trivial` · `modular` · `epic`).
2. Ломает существующих потребителей → **`major`**.
3. Иначе добавляет функциональность → **`minor`**.
4. Иначе совместимый багфикс → **`patch`**.

Причина ≠ вес: багфикс, который сам ломает совместимость, — **`major`**. Вес не выводится из BRD →
**STOP**, вопрос оператору. `level=epic` (>2 модулей или >1 репо) → **STOP**: алгоритм эпика не
реализован.

**Единственная механическая ревизия веса:** на `minor` диффер контрактов
(`validate-contract-diff.mjs --require-additive`) нашёл breaking-класс ⇒ вес был неверен ⇒ STOP и
ре-триаж как `major`.

## 5. Вертикали

| Вес | Голова конвейера | Транк | Тоггл | Тесты |
|---|---|---|---|---|
| **greenfield** | FRD → срезы → use-case → **заморозка контракта** → дерево модулей → README → тикеты → scaffold | `1.0.0` | новое OFF | юниты по формуле + полный компонентный набор |
| **patch** | `change-delta` + discriminating; **контракт не трогается**; дизайн по радиусу ряби | `Z+1` | обычно нет | discriminating old ≠ new там, где различие наблюдаемо; весь сьют зелёный |
| **minor** | аддитивная эволюция контракта (0 breaking); дизайн `needed` всегда | `Y+1.0` | **новое OFF обязательно** | компонентный на **новое** API (RED-first); существующие контрактные тесты не правятся |
| **major** | редизайн + миграция/депрекейшн; breaking-список в тело PR | `X+1.0.0` | тоггл + миграционный путь | формула держится; изменённые компонентные дорабатываются под новый контракт |
| **chore** | `docs/chores/<NNN-slug>/CHORE-PLAN.md`: файлы, команда верификации, откат | **no-bump** | — | приёмка = команда верификации из плана |

Общий хребет всех вертикалей: **Gate #1 → ветка от свежего транка → реализация по тикетам → DoD
(`@fagan`) → терминальный git-шаг (PR + CI) → Gate #2 → закрытие прогона → канарейка → Gate #3.**

## 6. Поток (happy path)

```
[Требование]
   │
   ▼ @gilb — измеримый BRD (открытые вопросы → оператору по одному)
   ▼ @wirth-triage — ВЕС (.agent/planner/mode)
   │
   ├─ chore ─────► CHORE-PLAN.md
   ├─ greenfield ► FRD → срезы → use-case → КОНТРАКТ (freeze) → модули → README
   └─ patch/minor/major ► change-delta (discriminating) → [контракт: minor/major] → дизайн, если needed
   │
   ▼ @wirth-ticketer → тикеты (тип + blocked_by) → @wirth-planner → PLAN.md
   ▼ @mills — ревью плана одним проходом (blocker → @linger → перезапуск)
   ▼ [HUMAN GATE #1] — токен GATE1 APPROVE (маркер ставит гардрейл, не агент)
   ▼ @git-hand mode=start — ветка <type>/<slug> от свежего транка (на транке код не пишется)
   ▼ реализация по одному тикету: @scaffolder / @wirth-tester (RED) / @hughes | @hughes-rework
   ▼ validate-layout (раскладка) → @fagan — DoD + инвариант веса, снятие @wip
   ▼ @git-hand mode=terminal — commit → push → PR (вес в заголовке) → CI
   ▼ [HUMAN GATE #2] — токен GATE2 APPROVE, оператор мержит
   ▼ @ledger → close-run.mjs: пруф мерджа → тег → docs/changes/LEDGER.md → вайп .agent/
   │        └─ no-bump → прогон закрыт, канарейки нет
   ▼ @michtom — канарейка 1% → 5% → 25% → 100% + 4 золотых сигнала
   ▼ [HUMAN GATE #3] — приёмка прод-релиза
```

## 7. Человеческие гейты

| Гейт | Что принимается | Механика |
|---|---|---|
| **#1** | план (`PLAN.md` / `CHORE-PLAN.md`) | оператор выдаёт **`GATE1 APPROVE`**; гардрейл ставит `.agent/gates/gate1.approved`. До маркера имплементаторы (`@hughes`, `@wirth-tester`, `@scaffolder`, `@hughes-rework`) жёстко заблокированы |
| **#2** | мерж PR (после `ci=green`) | оператор выдаёт **`GATE2 APPROVE`**; маркер `gate2.approved`; без него `close-run.mjs` отказывает |
| **#3** | прод-релиз после канарейки | вердикт по 4 золотым сигналам на 100% |

Маркеры ставит **только гардрейл** по точному токену — самоакцепт роли/роутера запрещён и блокируется
механически. Свободные слова («ок», «принято») акцептом не являются.

## 8. Закрытие прогона (RUN-CLOSE)

`.agent/` истинно, пока прогон **открыт**. После Gate #2 `@ledger` вызывает
`node harness/close-run.mjs --pr <N>` — три акта в причинном порядке (вайп последний, потому что
первые два читают то, что он стирает):

1. **Пруф мерджа** — PR влит (`state == MERGED`, читается у forge). Не влит → отказ, тега нет.
2. **Тег** — вес из `.agent/planner/mode`, арифметика в `ci/semver-bump.mjs`: `patch → Z+1`,
   `minor → Y+1.0`, `major → X+1.0.0`, `greenfield → 1.0.0`. Список тегов берётся из
   `git ls-remote origin` (локальные отстают); **форма тега** — от последнего релизного тега репо,
   при отсутствии тегов — дефолт `v`. Существующий тег не перезаписывается (`TAG_EXISTS`).
3. **Запись** в `docs/changes/LEDGER.md` (append-only, самодостаточная) → **атомарный вайп**
   `.agent/`; `decisions.log` сохраняется (это трасса, а не состояние).

- **`no-bump` — нормальный исход**, а не отказ: дифф трогает только плумбинг (`.github/`, `*.md`,
  `docs/`, `Dockerfile`, lock-файлы) → тега нет, канарейки нет, прогон всё равно закрыт.
- **Тег ставит только `@ledger`.** `ci/recipes/*` — образец для репозиториев **без** харнеса; в
  `template/` воркфлоу тегирования не кладётся: два источника тега недопустимы.
- Без явного закрытия `gate1.approved` прошлой задачи пропустил бы следующую через гейт.

## 9. Ralph Loop и его ограничители

Цикл «реализация → валидатор/DoD/CI → классификация → фикс» замыкается до решения задачи:

1. **Предохранитель K=2.** Имплементатор своё красное **не чинит** — чинит `@linger`; не зелено за
   две попытки → эскалация оператору (потолок держит `rational-guardrail`).
2. **Запрет на gaming тестов.** Агент не имеет права менять тесты, ассерты, CI-конфиги и пороги ради
   «позеленения». На `minor` правка существующего контрактного теста — **блокер** (сигнал неверного веса).
3. **Классификация дефекта обязательна:** дефект реализации → локальный фикс; дефект плана/шаблона →
   `escalate`. Решение логируется.
4. **Бюджет и step-cap.** Лимит шагов роли + бюджет токенов; исчерпание → эскалация.
5. **Стоп-условие.** Успех — только `@fagan accepted` + зелёный CI на PR. Иначе эскалация, не молчаливое
   завершение.

**Трассируемость:** каждое действие логируется в `.agent/decisions.log` — какая роль, какая модель,
какое решение и на основании чего. Путь «требование → план → код → PR → тег → прод» восстанавливается
по истории.

## 11. Что встроено (не потерять)

- **Компонентные тесты + математическая композиция** — доказательная корректность по построению;
  юниты по формуле `1 happy + Σ ветвей antecedent`, компонентные — `N = 1 + Σ различимых ветвей адаптера`.
- **Контрактные тесты** между компонентами (проверяет `pinout`) — контракт как артефакт плана, до имплементации.
- **Фиче-тогглы** — новая способность едет **выключенной**; тоггл же служит ручкой канареечной раскатки
  и мгновенного отката без передеплоя.
- **Discriminating-сценарий** — изменение обязано быть доказано old ≠ new на данных (для `minor` —
  отсутствие → присутствие); если ничто детерминированное не пиннит различие, причина названа в DoD.
- **Сетевая связанность для security** — новые сетевые проходы документируются в `network-topology.md`.
- **Приёмка прод-релиза после канарейки** (Gate #3) — то, что не покрыто автотестами, проверяется
  реальным трафиком под канарейкой.

## 12. Риски эксплуатации — за чем следить

1. Деградация ревью, если генератор и оценщик — одна модель. Держать асимметрию.
2. «Тихий» подгон под зелёное. Защита — запрет правок тестов + механические валидаторы.
3. Неверный вес (инфляция `patch`→`minor` или недооценка слома). Защита — диффер контрактов и
   ре-триаж, а не суждение на глаз.
4. «Слепой» постмониторинг: нет данных ≠ «зелёно» — расширять канарейку запрещено.
5. Зарастание тогглами. Защита — обязательный cleanup-тикет после 100%.
6. Незакрытый прогон: состояние прошлой задачи маскируется под новую. Защита — обязательный RUN-CLOSE.

## 13. Инженер-дирижёр (Engineer-Orchestrator)

Сквозная человеческая роль, которая ведёт задачу через весь Delivery. Это не «вайб-разработчик» —
наоборот, фигура, делающая дробление на узких «вайб-специалистов» ненужным.

### Зачем роль существует

Контроль переносится с *кода* на *границы*: на намерение (план) и на результат (ревью + CI + тег).
Кто-то должен стоять на этих границах и принимать решения — иначе human gates висят «в воздухе».

### Что человек контролирует

- **Контролирует:** постановку задачи, план, архитектурные решения, прохождение проверок, мерж и раскатку.
- **Больше не контролирует построчно:** имплементацию. Отсюда смещение навыка: критично умение
  **читать** код, в том числе на не до конца знакомом стеке.

### Зоны ответственности по стадиям

- **На входе.** Решает, что требование готово; ответы на открытые вопросы `@gilb` — его.
- **Gate #1 — акцепт плана.** Читает план, проверенный `@mills`: вписывается ли решение в архитектуру,
  не упущены ли НФТ, разумна ли декомпозиция и соответствует ли покрытие весу.
- **Во время цикла — разбор эскалаций** (K=2 исчерпан, `escalate` от `@mills`/`@linger`, `STOP` роли).
- **Gate #2 — мерж.** Читает финальный PR, выдаёт `GATE2 APPROVE`, мержит; дальше `@ledger` закрывает прогон.
- **Во время канарейки — разбор аномалий** (RED от `@michtom`): дефект реализации, конфига деплоя,
  неучтённое НФТ или шум метрик.
- **Gate #3 — приёмка прод-релиза** после стабильного прохождения канарейки до 100%.

### Требуемые компетенции

Архитектура и проектирование (главная, работает на Gate #1) · чтение кода · ясная постановка задачи ·
работа с группой агентов · многозадачность с фокусом · понимание принципов работы системы.

## 14. Agent-ready архитектура как входное требование

Ограничение контекста фундаментально. Единица работы агента — то, что помещается в контекст вместе со
всем нужным для корректного изменения: код модуля, его контракты, тесты, правила. Полный набор
требований — в скилле `architecture`. Встраивание в процесс:

- **Входной критерий.** Задача укладывается в границы 1–2 модулей; шире — сначала декомпозиция
  (триаж честно вернёт `epic` → STOP).
- **Обязанность планирования.** Дизайнер удерживает изменение в границах модулей (Parnas: один секрет);
  `@mills` это проверяет.
- **Долговременный долг.** Часто меняющиеся модули с дырявыми границами — первые кандидаты на декомпозицию.

## 15. Continuous Delivery и постмониторинг

Раздел описывает то, что происходит **после** закрытия прогона с тегом: как `@michtom` доводит
принятый код до работающей фичи в проде.

### 15.1 Канареечный релиз прямо в прод

Зелёный CI доказывает корректность кода в изоляции. Он **не** доказывает, что фича установилась,
поднялась и здорова под реальным трафиком. Стендов нет — артефакт едет **канарейкой прямо в прод за
фиче-тогглом** на вариативную среду (VM / контейнер / serverless, **не обязательно Kubernetes**):

| | КАНАРЕЕЧНЫЙ РЕЛИЗ В ПРОД |
|---|---|
| **Цель** | фича здорова под реальным трафиком |
| **Трафик** | реальный, пользовательский |
| **Тоггл** | OFF на старте, открывается канарейкой |
| **Раскатка** | доля 1% → 5% → 25% → 100% |
| **Главный сигнал** | 4 золотых сигнала + SLO + guardrail vs baseline |
| **Кто оценивает** | `@michtom`: фаза анализа выносит GREEN/YELLOW/RED |
| **Gate** | Human Gate #3 — приёмка прод-релиза после канарейки |
| **Откат** | тоггл OFF без передеплоя |
| **Когда НЕ запускается** | `no-bump` (chore/плумбинг): катить нечего |

### 15.2 Шаги канареечного релиза

1. **Артефакт версии тега.** Тег на транке поставил `@ledger`; фича закрыта тогглом (OFF).
2. **Деплой в прод с тогглом OFF.** Проверяется здоровье окружения с новой версией — сама выкатка не
   должна ничего сломать ещё до включения фичи.
3. **Канареечная раскатка.** Тоггл открывается ступенями (1% → 5% → 25% → 100%), на каждой доле — окно наблюдения.
4. **Постмониторинг доли.** 4 золотых сигнала (latency / traffic / errors / saturation) + SLI фичи +
   бизнес-guardrail против baseline: **GREEN** → расширить · **YELLOW** → держать долю · **RED** →
   немедленный откат (тоггл OFF) + эскалация.
5. **Human Gate #3 / fast-path.** Выход на 100% значимых фич подтверждает дирижёр; мелкие с устойчиво
   зелёными долями доезжают автоматически.
6. **Стабилизация и уборка.** После 100% фича принята; тоггл — техдолг, заводится cleanup-тикет.

### 15.3 Rollout Loop и его ограничители

1. **Только вперёд по зелёному.** YELLOW или нехватка окна — не расширять.
2. **Откат дешевле починки.** При RED сначала безопасное состояние (тоггл OFF), потом разбор.
3. **Запрет на gaming сигналов.** Нельзя ослаблять SLO, менять определения SLI, глушить алерты или
   трактовать отсутствие данных как «зелёно» → автооткат и эскалация.
4. **Error budget как стоп-условие.**
5. **Лимит времени наблюдения:** зависание в YELLOW → эскалация дирижёру.

### 15.4 Трассируемость CD

Каждый шаг выката логируется в `decisions.log` и `release-health.md`: версия/тег, доля канарейки,
сигналы и baseline, окно, вердикт, был ли откат и почему. Путь «код → тег → прод → раскатка →
результат» должен полностью восстанавливаться по истории.
