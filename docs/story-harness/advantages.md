# Достоинства скиллов проектирования и разработки

Почему этот набор скиллов (и собранный из них харнес) — **рациональный
научно-обоснованный подход**, а не ещё один промпт-пак. Кратко: качество достигается
**проектированием по доказанной дисциплине**, а тесты — это **рациональный минимум**
(юнит по формуле + компонентные + контрактные), не «сеть для багов» и не покрытие ради покрытия.

## TL;DR

| Принцип | Что даёт |
|---|---|
| Корректность **по построению** (Cleanroom, Mills) | дефект не ищется отладкой — он не вносится |
| Программа = **дерево модулей** (Wirth), композиция по контрактам | сложность дробится на проверяемые до старта куски |
| Тесты — **рациональный минимум** (формула, чёрный ящик, контракт) | максимум уверенности при минимуме машинных ресурсов |
| Скилл = **процедура** (шаги, таблицы-роутеры, STOP), не справочник | работает даже на слабой модели (~17B), детерминированно |

## 1. Научная база проектирования

Подход опирается не на моду, а на труды инженеров структурного программирования —
фундамент, проверенный десятилетиями:

| Труд | Авторы | Что взято |
|---|---|---|
| *Systematic Programming: An Introduction* | Niklaus **Wirth** | программа = дерево модулей; декомпозиция сверху вниз; пошаговое уточнение; минимализм |
| *A Structured Approach to Programming* | Joan **Hughes**, Jay **Michtom** | прикладное структурное кодирование |
| *Structured Programming: Theory and Practice* | Richard **Linger**, Harlan **Mills**, Bernard **Witt** (IBM) | Cleanroom, корректность по построению, функциональная верификация |

Ключевые идеи, вшитые в скилл `program-design`:

- **корректность по построению, а не «отладка тестами»** (Cleanroom Миллса);
- **математическая композиция модулей**: предусловие/постусловие (антецедент/консеквент),
  один data-аргумент, `Result<T,E>` — модуль защищён на входе и выходе;
- **бизнес-логика отделена от I/O** — структурная корректность как инвариант;
- **вертикальные срезы** — единица работы помещается в контекст (agent-ready);
- **чёрный ящик**: вызывающему нужны имя модуля и контракт, не внутренности.

> «Качество достигается проектированием, а не тестированием.» Отсюда и рациональный,
> а не раздутый, набор тестов ниже.

## 2. Рациональный набор тестов

Не «покрыть всё», а **проверить то, что важно, минимумом ресурсов**. Три уровня, у
каждого своя работа — они не дублируют друг друга:

| Уровень | Скилл | Что проверяет | Принцип рациональности |
|---|---|---|---|
| **Юнит — по формуле** | `program-design` (юнит-тесты по формуле) | граничные значения, happy path, **задокументированные** коды ошибок модуля | ровно столько, сколько вытекает из контракта модуля — не больше |
| **Компонентные — чёрный ящик** | `component-tests` | спецификацию интерфейса программы: штатный путь + отказ **каждой** внешней связи | `N = N_эндпоинтов + Σ режимов_отказа`; контракт первичен, не код |
| **Контрактные — между сервисами** | `contract-tests` / [`pinout`](https://github.com/codemonstersteam/pinout) | совместимость связей (netlist контрактов) | проверяется до прода, отдельным проектом |

Свойства этого набора:

- **Контракт первичен.** Компонентные тесты пишутся **до кода** (TDD), режимы отказа
  берутся из OpenAPI/AsyncAPI и «Карты режимов отказа» README — не выводятся из адаптеров.
- **Исполнимая спецификация.** Тест на Gherkin — одновременно спека и автотест; сервис
  проходит его, ничего не зная о своих внутренностях.
- **Изоляция единообразна** — вся среда в Docker Compose (сервис/CLI + заглушки внешних API).
- **Юнит — по формуле, не по интуиции.** Набор юнит-тестов выводится из контракта модуля
  (границы, happy path, документированные ошибки), поэтому он конечен и обоснован.

## 3. Почему это выгодно

- **Предсказуемость и воспроизводимость.** Дисциплина одинакова от задачи к задаче;
  без единого фрейма агенты импровизируют и теряют трассируемость.
- **Работает на слабой модели.** Скиллы — пошаговые **процедуры** (шаги, таблицы-роутеры,
  чеклисты, STOP-правила), рассчитанные на ~17B активных параметров: модели не нужен
  «гений», нужна процедура.
- **Минимум машинных ресурсов.** Тесты — рациональный минимум, корректность — композицией;
  не жжём CI и токены на покрытие ради покрытия.
- **Трассируемость инженерная, не бюрократическая.** Решения → `decisions.log`, рабочий
  контекст → `.agent/memory.md` (skill `memory`); путь «план → код → релиз» восстановим.
- **Качество смещено влево.** Дефект дешевле не внести (проектирование), чем поймать тестом.

## Итог

Это не «ускорение печатания кода», а **перенос инженерной дисциплины структурного
программирования в агентов**: проектируем по Вирту/Миллсу, доказываем корректность
композицией, а тестами фиксируем спецификацию — рациональным минимумом. Поэтому харнес
из этих скиллов даёт воспроизводимый результат даже на недорогих моделях.
