# Intent — превратить репозиторий в эталон AI-first SDLC

> Пример структурированного промпта: бриф инженера агенту на рефакторинг репозитория.

## Цель

Превратить текущий репозиторий в эталон AI-first разработки: описать SDLC для
человека и сгруппировать существующие скиллы по агентским модулям согласно процессу —
чтобы на каждом шаге было сразу понятно, что применять.

## Источники истины

- `00_PROCESS.md` — целевой процесс: роли, гейты, поток (happy path).
- `skills/` — существующие скиллы: реализация сервиса и документирование продукта (распределённая система и отдельный сервис).
- `docs/story/` — описание агентских модулей.

## Задачи

1. Проанализировать репозиторий.
2. В основном README лаконично описать SDLC по `00_PROCESS.md`.
3. Разложить скиллы по агентским модулям (роль = модуль): для каждого шага процесса видно, что грузит агент.
   - Пример (§4 «Поток (happy path)»): внутри рамки планировщика — `architecture / code-style / security / observability` → пишет `plan.md`.
4. Подсветить, чего не хватает для полного функционирования процесса.

## Рабочая модель

Инженер даёт задачу планировщику на вход; тот работает по своим скиллам,
сгруппированным в его модуле. И так — по всему процессу.

## Масштабирование по объёму задачи

- **Эпик** — работа со всей полнотой процесса.
- **Стори / фича в сервисе** — какой агентный модуль использовать и что ему на вход (план разработки фичи).
- Где хранятся артефакты по SDLC: ПЛАН — в `backlog.md` метарепозитория и в сервисе; передаётся с конкретным пакетом проектировщика фичи (см. `skills/`).

## Метод

1. Сначала — тезисный план работ.
2. После пруфа — описать его в `plan.md`.
3. Далее — рефакторинг проекта по плану.

## Результат (Definition of Done)

Чёткая картина: какие артефакты нужны каждому агент-модулю на вход и как он
валидирует их полноту (контракт вход/выход) — чтобы более слабая модель могла
реализовать всё по соответствующим документам.
