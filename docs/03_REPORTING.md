# Система отчётности по применению ИИ и бэклог подразделений

> Версия 1.0. Документ (а) описывает **систему отчётности** — какие отчёты, для кого и в каком окне снимаются; (б) служит **источником бэклога**: рабочие пакеты по подразделениям, выводимые из измерительной спеки.
>
> Разделение ответственности с соседним документом:
> - [`02_MEASUREMENT.md`](02_MEASUREMENT.md) — **что и как** считаем (определения метрик, механика съёма, формулы).
> - **Этот документ** — **когда и кому** отчитываемся (окна, аудитории, артефакты) и **кто что делает**, чтобы метрики поехали.

## 1. Система отчётности

### 1.1 Уровни и окна

Окно отчётности — не одно. Разные потребители принимают решения с разной частотой, поэтому метрики первой очереди ([`02`](02_MEASUREMENT.md)) раскладываются на три уровня:

| Уровень | Аудитория | Окно агрегации | Каденс публикации | Метрики | Артефакт |
|---|---|---|---|---|---|
| **Операционный** | команда / инженер-дирижёр | скользящая неделя | еженедельно | AI-commit %, AI-PR % (`ai-authored` vs `ai-assisted`), Frame-drift count | дашборд команды |
| **Тактический** | тимлид | спринт (2 недели) | раз в спринт | Prod AI-deploy % (+ `rolled-back`, stable@100%), Frame-capture coverage | спринт-ревью отчёт |
| **Стратегический** | основатели / тех-лид | месяц / квартал | ежемесячно | тренды адопшена и эффективности (вторая очередь [`02 §7`](02_MEASUREMENT.md)), Frame-capture coverage как трассируемость | борд руководства |

### 1.2 Окно по каждой метрике

| Метрика | Окно агрегации | Каденс | Уровень |
|---|---|---|---|
| AI-commit % / AI-PR % | скользящие 7 дней | еженедельно | операционный |
| Prod AI-deploy % (+ `rolled-back`) | спринт | раз в спринт | тактический |
| Frame-capture coverage | спринт | раз в спринт | тактический + страт. (трассируемость) |
| Frame-drift count | реальное время | алерт сразу + недельная сводка | операционный |

### 1.3 Принципы отчётности

- **Окно агрегации ≠ каденс публикации.** Скользящая неделя может публиковаться ежедневно; спринтовое окно — раз в спринт. Не путать.
- **Всегда показываем знаменатель.** «42% AI-PR» бессмысленно без «от скольких PR». Без знаменателя метрика не публикуется.
- **Никакой агрегации на инженера** как продуктивности (правило [`02 §2`](02_MEASUREMENT.md)). Разрез — репозиторий / команда / поток.
- **Только из источников истины:** git, `decisions.log`, релизные артефакты ([`02 §6`](02_MEASUREMENT.md)). Ручная разметка в отчётность не попадает.
- **Frame-drift и `rolled-back` для AI-кода** — не агрегируются в «процент», а показываются абсолютным числом с трассировкой: это сигналы инцидента, а не тренд.

## 2. Бэклог по подразделениям

Рабочие пакеты, без которых метрики [`02`](02_MEASUREMENT.md) и отчётность §1 не поедут. Каждый пакет: что сделать, какую метрику разблокирует, Definition of Done, зависимости. Подразделения тянут задачи отсюда.

| P-1 | Реестр агентских идентичностей (роль → email `@agents.local` → модель), версионируемый в git | М1, М2 | реестр в git, счётные скрипты опираются только на него | — |

### 2.6 Метрики ускорения производства (отдельный трек — на проработку)

> **Отдельный трек, вынесен сознательно.** Метрики внедрения (М1–М3) отвечают на «сколько ИИ и прослеживаемо ли это». Метрики ниже отвечают на другой вопрос — «**стало ли быстрее и дешевле производить ПО**». Это вторая очередь ([`02 §7`](02_MEASUREMENT.md)) и связано с рисками RM-3/RM-4/RM-5/RM-7 (§5.2). Их нельзя считать достоверно без бейзлайна и методологии атрибуции, поэтому они здесь как **пакет на проработку**, а не на немедленное внедрение. Презентации руководству пока **не показываем как обещание** — только после закрытия EFF-0.

| ID | Рабочий пакет | Измеряет | DoD | Зависит от |
|---|---|---|---|---|
| EFF-0 | **Бейзлайн + методология атрибуции** (бейзлайн до масштабирования, staggered rollout / контрольная когорта) | предусловие для всех EFF | зафиксированный бейзлайн «до», описанная процедура атрибуции «агенты → эффект» | RM-3 |
| EFF-1 | Cycle-time по стадиям Delivery (план → мерж → канарейка → прод) | скорость такта | время по стадиям считается из `decisions.log` / git за окно, со знаменателем | EFF-0, R-1, P-4 |
| EFF-2 | Throughput и параллелизм на инженера-дирижёра (задач/спринт, одновременно ведомых задач) | пропускная способность | значения за окно, парный guardrail — DevEx-пульс (когнитивная нагрузка), §5.4 | EFF-0 |
| EFF-3 | Escalation rate и число итераций цикла (Ralph Loop / Rollout Loop) | «цена» автономии | доля задач с эскалацией и распределение итераций за окно | EFF-0 |
| EFF-4 | Экономика: `cost-per-merged-PR`, `tokens/задачу` (p50/p95), GPU-секунды/исход | стоимость единицы результата (RM-5) | стоимость на смерженный PR и на задачу, с распределением | EFF-0, D-1 |
| EFF-5 | Speedup vs бейзлайн: lead time агентский vs ручной по сопоставимым задачам | бизнес-импакт (RM-7) | сравнение с бейзлайном/когортой, помечено lead/lag | EFF-0, EFF-1 |

**Почему отдельно:** ускорение — это **исход**, а внедрение (М1–М3) — **факт применения**. Смешивать их в одной отчётности опасно: «80% адопшена» легко прочитать как «на 80% быстрее», что неверно. Метрики ускорения требуют контрольной базы и не публикуются как тренд до EFF-0.

## 3. Очередность (волны)

Логика: **сначала источники истины, потом коллекторы, потом дашборды.** Считать то, что ещё не пишется в источник, нельзя.

1. **Волна 1 — источники истины:** P-1, P-2, P-3, P-4, R-1, G-1. Без них считать нечего.
2. **Волна 2 — анти-gaming и коллекторы:** G-2, G-3, R-2, D-3, P-5.
3. **Волна 3 — витрина, отчётность, трассируемость:** D-1, D-2, C-1, C-2.

> **Трек «ускорение производства» (§2.6, EFF-0…5)** идёт параллельно волнам и не блокирует пилот: его предусловие — бейзлайн (EFF-0) — снимается **до старта пилота**, иначе атрибуция эффекта будет невозможна задним числом.

## 4. Definition of Done для метрики

Метрика считается «в проде отчётности», когда выполнены все четыре условия:

- [ ] **Источник истины пишется** — признак/фрейм фиксируется в git / `decisions.log` / артефакте автоматически.
- [ ] **Коллектор считает** — скрипт даёт значение со знаменателем и трассировкой за заданное окно.
- [ ] **Дашборд показывает** — метрика на нужном уровне отчётности (§1.1) в нужном окне.
- [ ] **Анти-gaming активен** — подмена/снятие признака детектируется и блокируется.

## 5. Риски системы измерения и пробелы относительно мировых практик (вход для анализа)

> Раздел фиксирует пробелы первой очереди измерения относительно мировых практик (DORA, SPACE, DevEx, DX Core 4, гайды GitHub по измерению AI-адопшена, закон Гудхарта). Это **не выводы, а риск-реестр** — вход для дальнейшего анализа и приоритизации переноса метрик между волнами.

### 5.1 Вердикт

Первая очередь меряет **«сколько ИИ»** (penetration, М1/М2) и **«прослеживаемо ли это»** (provenance/governance, М3). Она почти не меряет **«хорошо ли это»** (value/quality) и **«лучше ли стало людям»** (experience). По зрелости provenance/governance система **впереди** рынка; по балансу метрик — **перекошена в учёт, а не в исход**. Корень — граница wave1/wave2 ([`§7 в 02`](02_MEASUREMENT.md)) нарезала систему так, что вся ценность/качество/опыт/стоимость уехали во вторую волну.

### 5.2 Риск-реестр

| ID | Пробел / риск | Измерение (DX Core 4) | Практика-ориентир | Последствие | Предлагаемое действие | Статус |
|---|---|---|---|---|---|---|
| RM-1 | Нет качества/исхода во wave 1; единственный качественный сигнал — `rolled-back` (редкий, запаздывающий) | Quality | SPACE, DX Core 4, GitHub | «80% адопшена» при молчаливой деградации качества | Поднять во wave 1: CFR (AI vs human), rework/число итераций Ralph Loop | открыт |
| RM-2 | Нулевой перцептивный слой (нет self-report) | Experience | SPACE, DevEx | Перегруз/недоверие дирижёра не видны до оттока | Лёгкий пульс-опрос дирижёров во wave 1 (доверие, когнитивная нагрузка, perceived productivity) | открыт |
| RM-3 | Нет методологии атрибуции/бейзлайна | сквозное | quasi-experiment | Нельзя доказать причинность «агенты → улучшение» | Зафиксировать бейзлайн до масштабирования + staggered rollout / контрольная когорта | открыт |
| RM-4 | М1 — penetration ≠ depth; насыщается к ~100% | Adoption depth | GitHub | После насыщения М1 неинформативна | Добавить autonomy/override rate на Human Gate #1 и #2, escalation rate | открыт |
| RM-5 | Нет экономики (cost/efficiency); GPU-ограничение не измеряется | Cost / efficiency | FinOps for AI | Главное ограничение (9×H100) вне отчётности | `cost-per-merged-PR`, `tokens/задачу` (p50/p95) во wave 1 | открыт |
| RM-6 | Метрики не спарены с counter-metrics (анти-gaming как правила, не как парные показатели) | сквозное | Goodhart / Campbell | Рост первичной метрики гeймится без видимости | Balanced scorecard с guardrail-парами (§5.4) | открыт |
| RM-7 | Нет деления lead/lag и нет бизнес-импакта | Business impact | DX Core 4 | Отчёт нельзя читать как раннее предупреждение; нет связи с ценностью | Пометить метрики lead/lag + назвать ≥1 value-показатель | открыт |
| RM-8 | Не измеряется надёжность самого инструментирования | Data quality | «мониторь монитор» | Все метрики стоят на необеспеченном фундаменте | Метрика: % коммитов с парсимым trailer, полнота `decisions.log` | открыт |

### 5.3 Balanced scorecard: текущий баланс wave 1

Минимально сбалансированный wave 1 требует ≥1 метрики на каждое измерение. Текущее состояние:

| Измерение | Сейчас во wave 1 | Дефицит — поднять из wave 2 |
|---|---|---|
| Adoption / penetration | М1, М2 ✅ | autonomy/override rate (глубина) |
| Quality | только `rolled-back` ⚠️ | CFR (AI vs human), rework/итерации |
| Experience | нет ❌ | пульс-опрос дирижёров |
| Cost / efficiency | нет ❌ | cost-per-merged-PR, tokens/задачу |
| Governance | М3 ✅ | — (сильная сторона) |
| Attribution | нет ❌ | бейзлайн + контрольная когорта |

### 5.4 Парные counter-metrics (против Гудхарта)

К каждой первичной метрике — явный guardrail её провального режима; рост первичной читается только вместе с контр-показателем.

| Первичная | Режим провала при оптимизации | Guardrail (counter-metric) |
|---|---|---|
| М1 AI-PR % | гонят объём через агента в ущерб глубине ревью | rework/override rate, CFR (AI) |
| М2 Prod AI-deploy % | агрессивно катят AI-код | `rolled-back`, CFR (AI), escaped defects |
| М3 Frame coverage | реклассификация действий в «исключения»/вне фрейма | Frame-drift count, exception-rate |
| throughput / дирижёр (wave 2) | перегруз дирижёра | DevEx-пульс (когнитивная нагрузка), escalation rate |

### 5.5 Что сделать с этим разделом

- Риски RM-1…RM-8 — кандидаты на перенос соответствующих пакетов из второй волны ([`§7 в 02`](02_MEASUREMENT.md)) в первую волну (§3) после анализа и приоритизации.
- Provenance/governance (М3, §2.1) — **не трогать**, это сильная сторона; не вкладываться в его углубление в ущерб закрытию RM-1/RM-2/RM-5.
- Статусы в §5.2 ведутся как живой реестр: `открыт → в анализе → принят в wave 1 / отклонён (с обоснованием)`.
