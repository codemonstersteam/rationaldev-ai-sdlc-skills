# Этапный конвейер планирования (staged planning pipeline)

> Как задача на разработку сервиса (модуля системы) разбивается на атомарные этапы,
> каждый из которых — отдельный тикет со своим контекстом. Версия 0.1 (draft).
> Дополняет [`00_PROCESS.md`](00_PROCESS.md): раскрывает стадию планирования Delivery.

## 1. Зачем

Монолитное «планирование» не сходится на слабых моделях: имплементер затыкается, контекст
не влезает, документации/ADR нет. Решение — **разбить планирование на цепочку атомарных
этапов**. Каждый этап:

- отдельный тикет со своим **самодостаточным контекстом**;
- контекст **влезает в целевую модель** (планирование — GLM 5.2, имплементация — Qwen3.6-27b);
- один этап = одна ответственность = один скилл;
- решения приняты **детерминированно** (таблицы, формулы), а не «на усмотрение» модели.

Правило: **выбор скилла — не суждение модели, а lookup по таблице** (см. §4).

## 2. Целевые модели

| Стадия | Модель | Что делает |
|--------|--------|-----------|
| Планирование / проектирование | **GLM 5.2** | этапы 0–11: спеки, дерево модулей, нарезка тикетов |
| Имплементация тикета | **Qwen3.6-27b** | один тикет = один модуль/срез: тесты → модуль → зелёное |

Каждый артефакт этапа обязан помещаться в контекст своей модели вместе с нужными скиллами.

## 3. Конвейер: этап → артефакт → скилл

Где физически лежит каждый артефакт (раскладка репозитория, форматы `CONTEXT.md`/ADR) —
см. [`05_REPO_STRUCTURE.md`](05_REPO_STRUCTURE.md).

| # | Этап | Артефакт | Скилл | Где |
|---|------|----------|-------|-----|
| 0 | Уточнение требований + верхнеуровневый план | FRD, актёры, глоссарий, список слайсов | ♻️ `requirements-intake` | — |
| 1 | Нарезка на Vertical Slices | атомарные слайсы (end-to-end, acceptance, Blocked by) | 🆕 `vertical-slices` | — |
| 2 | Cockburn use case на сервис | fully-dressed use case | 🆕 `cockburn-use-case` | `src/<ctx>/docs/design/<slice>/use-case.md` |
| 3 | OpenAPI спека | контракт sync HTTP | 🆕 `openapi-spec` | `src/<ctx>/api-specification/openapi.yaml` |
| 4 | AsyncAPI спека | контракт async | 🆕 `asyncapi-spec` | `src/<ctx>/api-specification/asyncapi.yaml` |
| 5 | Компонентные тесты по спеке | `.feature` по формуле, RED-ready | ♻️ `component-tests` | шаблон passkey |
| 6 | Тесты собираются и RED | скелет сервиса, все тесты красные по бизнес-причине | 🆕 `component-test-scaffold` | — |
| 7 | README.md | документация сервиса | ♻️ `documentation` | корень репо |
| 8 | STEP 1: дерево модулей + псевдокод головного | дерево + псевдокод → оператору (Gate #1) | ♻️ `program-design` | — |
| 9 | Карта контекста слайса | CONTEXT-MAP/CONTEXT.md, ADR, дерево, контракты, C4 + use case | ➕ `domain-modeling`-структура + 🆕 `c4` | `src/<ctx>/...` |
| 10 | STEP 2: проектирование каждого модуля | пакет модуля: контракт, границы, формула юнит-тестов, `io:` | ♻️ `program-design` + io-под-скилл | — |

## 4. Детерминированный io-роутинг (этапы 10–11)

Тип I/O модуля фиксирует **проектировщик** на этапе 10 — обязательное поле в контракте
каждого модуля:

```
io: none | http | llm | queue | db
```

`implementation-ticket-writer` (этап 11) применяет **фиксированную таблицу** тип → скиллы:

| `io` | Скиллы в контекст тикета |
|------|--------------------------|
| `none`  | `program-design` |
| `http`  | `program-design` + `http-io` |
| `llm`   | `program-design` + `http-io` + `llm-client` |
| `queue` | `program-design` + `queue-io` |
| `db`    | `program-design` + `db-io` |

Имплементер (Qwen) **ничего не выбирает** — получает тикет с уже вложенными нужными скиллами.

## 5. Разделение уровней тестирования (жёстко)

- **Компонентные тесты** (чёрный ящик сервиса): `N = 1 + Σ (различимых веток адаптера i)` —
  happy path + failure-ветки внешних зависимостей. Обычно 3–6 на сервис. Формулировки веток —
  дословно из расширений Cockburn use case (сквозная трассируемость).
- **Все границы, классы эквивалентности, decision tables → юнит-тесты модуля** (этап 10).
  Каждая граница покрыта юнит-тестом. **Ни одной границы в компонентных.**

## 6. Гейты и приёмка

- **Дизайн-гейт (этап 8) = часть Gate #1.** Оператору на Gate #1 кладут: декомпозицию (слайсы),
  **дерево модулей**, **псевдокод головного модуля**, контракты/спеки. Отдельного 4-го гейта нет.
- **Приёмка слайса.** Метку **`@wip`** на компонентных сценариях слайса **ставит этап 5**
  (авторинг тестов, RED) — это выходное требование тикета этапа 5, не отдельный тикет.
  По завершении разработки **всего слайса фиксер** запускает компонентные на слайс; при
  **GREEN снимает `@wip`** и принимает работу (снятие метки = акт приёмки, снимает только фиксер).
  Триггер — оркестратор видит последний тикет слайса зелёным (автопрогон до Gate #2).
  Цикл метки: **этап 5 ставит → фиксер снимает.**
- **Секвенция тестов:** сборка → юнит (per-module, каждый PR) → компонентные (только когда слайс
  собран целиком; до полной сборки они структурно красные — нет сигнала).

## 7. Результат конвейера

- Декомпозированный план работ (слайсы + тикеты, dependency-ordered).
- Спецификации: Cockburn use case, OpenAPI, AsyncAPI.
- Компонентные тесты (собираются, RED, помечены WIP).
- README.md.
- Карта контекста слайса: CONTEXT-MAP/CONTEXT.md, ADR, дерево модулей, контракты модулей +
  их совместимость, C4, рядом use case.
- Пакеты проектирования модулей (по одному на модуль).
- Тикеты реализации под Qwen3.6-27b (минимальный контекст).

Финал: субагенты собирают сервис по кирпичикам; фиксер прогоняет сборку → юнит → компонентные,
принимает слайс. **Перед делегированием независимых тикетов — сверка с оператором:** какой план,
тикеты, структура программы получились.
