# Структура репозитория сервиса

> Где живёт каждый артефакт конвейера планирования ([`04_PLANNING_PIPELINE.md`](04_PLANNING_PIPELINE.md))
> и как форматируются `CONTEXT.md` и ADR. Версия 0.1 (draft). Структура из DDD-практики
> (bounded contexts + ubiquitous language + ADR), расширенная контрактом сервиса и дизайном слайса.
> Файлы создаются **лениво** — только когда есть что записать.

## 1. Раскладка

**Один контекст (большинство сервисов):**

```
<repo>/
├── README.md                         ← documentation (этап 7)
├── CONTEXT.md                        ← глоссарий (§3)
├── api-specification/                ← ОДИН контракт на сервис
│   ├── openapi.yaml                  ← этап 3
│   └── asyncapi.yaml                 ← этап 4
├── docs/
│   ├── adr/                          ← решения уровня сервиса (§4)
│   │   ├── 0001-<slug>.md
│   │   └── 0002-<slug>.md
│   └── design/
│       └── <slice>/                  ← карта контекста слайса (этап 9)
│           ├── use-case.md           ← Cockburn fully-dressed (этап 2)
│           ├── c4.md                 ← C4 (этап 9)
│           ├── module-tree.md        ← дерево модулей + псевдокод головного (этап 8)
│           └── contracts.md          ← контракты модулей + их совместимость (этапы 9–10)
└── src/
```

**Несколько контекстов** — появляется корневой `CONTEXT-MAP.md`, а `CONTEXT.md`/`docs/adr`
переезжают внутрь контекста:

```
<repo>/
├── CONTEXT-MAP.md                    ← карта контекстов + связи (§3)
├── api-specification/                ← контракт сервиса (общий)
├── docs/adr/                         ← решения уровня системы
└── src/
    ├── ordering/
    │   ├── CONTEXT.md
    │   ├── docs/adr/                 ← решения контекста
    │   └── docs/design/<slice>/
    └── billing/
        ├── CONTEXT.md
        ├── docs/adr/
        └── docs/design/<slice>/
```

## 3. Формат `CONTEXT.md` (глоссарий)

Только термины — не спека, не блокнот, без деталей реализации.

```md
# <Имя контекста>

<Одно-два предложения: что это за контекст и зачем он.>

## Язык

**Order**:
Заявка клиента на покупку, принятая системой.
_Avoid_: Purchase, transaction

**Invoice**:
Запрос оплаты, выставленный клиенту после доставки.
_Avoid_: Bill, payment request
```

Правила: опинионейтед (один канонический термин, синонимы — в `_Avoid_`); определения tight
(1–2 предложения, что это ЕСТЬ, а не что делает); только термины **этого** контекста (общие
программистские концепты не включать); группировать под подзаголовки при естественных кластерах.

**`CONTEXT-MAP.md`** (при >1 контексте): секция `## Contexts` (список + ссылки + одна строка
назначения) и секция `## Relationships` (кто кому шлёт события / общие типы).

## 4. ADR

Живут в `docs/adr/`, sequential-нумерация `NNNN-slug.md`. Создавать лениво.
**Формат — [`templates/adr.md`](templates/adr.md)** (Статус / Контекст / Решение / Последствия);
здесь не дублируется. Реестр всех шаблонов — [`templates/README.md`](templates/README.md).

**Когда заводить ADR — все три условия истинны:**

1. **Трудно откатить** — цена смены решения потом значима.
2. **Неочевидно без контекста** — будущий читатель спросит «почему так?».
3. **Результат реального trade-off** — были альтернативы, выбрали одну по конкретным причинам.

Квалифицируется: архитектурная форма; паттерны интеграции между контекстами; выбор технологии
с lock-in (БД, брокер, auth); решения о границах/scope; намеренные отклонения от очевидного пути;
ограничения, не видимые в коде; неочевидно отклонённые альтернативы. Иначе — ADR не нужен.
