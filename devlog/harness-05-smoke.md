# devlog/harness-05-smoke.md

## Задача

Slice 5 харнеса: компонентный смоук — Gherkin-спецификация + исполняемая проверка
установки.

## Контекст и ограничения

- Установка детерминирована → `@auto` сценарии, прогоняются скриптом.
- Маршрутизация ролей требует живой модели → `@manual @model` (за оператором).
- Сервисный `component-tests/` (godog/Docker) — для сервисов-чёрных-ящиков; харнес не
  сервис, поэтому смоук кладём в `harness/smoke/` (отклонение от формулировки плана —
  обосновано: не смешивать с сервисными компонентными тестами).

## Принятое решение

- `harness/smoke/harness.feature` — 4 `@auto` (установка claude/opencode/codex + сборка
  Codex-AGENTS + недеструктивность) + 1 `@manual @model` (маршрутизация planner→reviewer→implementer).
- `harness/smoke/run.sh` — гоняет `install.sh` во временные каталоги, проверяет:
  раскладку (6 ролей, скилл memory), наличие инструкций, сборку Codex-AGENTS (orchestrator+Hughes),
  недеструктивность (оригинальный AGENTS.md цел, рядом `AGENTS.harness.md`). `mktemp` + cleanup.

## Результат

- `./harness/smoke/run.sh` → **PASS 11/11**.
- План харнеса (Slice 1–5) выполнен; остаётся опциональный Slice 6 (`--hard` enforcement).

## Отклонённые варианты

- godog/Docker-раннер для харнеса — избыточно: харнес не сетевой сервис, проверяем
  поведение `install.sh` напрямую на shell.
