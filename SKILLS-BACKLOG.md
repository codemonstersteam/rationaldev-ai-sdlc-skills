# Бэклог скиллов — что дописать для полного цикла

> Процесс ([`docs/00_PROCESS.md`](docs/00_PROCESS.md)) описан целиком, но часть ролей
> пока без детерминированных скиллов. Это **бэклог для доменных команд**: каждая
> команда пишет скилл по своему домену в стиле существующих `skills/lib/*`
> (таблицы-решения, числовые лимиты, STOP-правила, чеклисты; расчёт на слабую модель ~17B).
>
> Каждый скилл живёт в `skills/lib/<name>/SKILL.md` и подключается к роли-потребителю
> ссылкой из её манифеста `skills/roles/<role>/<role>.md`.

## Критерий готовности скилла (Definition of Done)

- [ ] Лежит в `skills/lib/<name>/`, подключён из манифеста роли-потребителя.
- [ ] Детерминированная процедура: таблицы-роутеры вместо суждений, числовые пороги, STOP-правила.
- [ ] Явный контракт вход/выход роли: какие `.agent/`-артефакты читает и пишет.
- [ ] Анти-gaming правила, где применимо (запрет ослаблять проверки ради «зелёного»).
- [ ] Проходит `doc-quality-review`.

## Очередь (по приоритету)

### 🔴 P0 — канареечный CD и безопасность

|---|---|---|---|---|

### 🟢 P1 — выразительность скиллов проектирования и документирования

Доработать `skills/lib/documentation` и `skills/lib/program-design` в соответствии с вводными ниже. Источник — результаты сессии `pinout-openapi`; референс-образцы готовы: use case по Коберну и C2/C3 — [`c4.md`](https://github.com/codemonstersteam/pinout-openapi/blob/main/docs/design/contract-validate/c4.md), C1 — [концепт pinout](https://github.com/codemonstersteam/pinout), модель ошибок — `messages.md`/`risk-coverage.md`, README по скиллу.

**Статус: ✅ реализовано в скиллах.** Все строки ниже внесены в процедуры:
- `documentation`: роутер (C4-диаграммы, use case, таблица сбоев); Pass A5b (таблица сбоев + модель ошибок); Pass B3 (C4 по уровням); Pass B4 (use case по Коберну); STOP-правило + финальный чеклист (doc-gate).
- `program-design`: Шаг 3 (единый `Request` + развилка=логика, D1; «C4 по уровням»); Шаг 4 (модель ошибок); Шаг 8.1 (развилки в юнитах) + Шаг 8.6 (трассировка UC→слайс→тест); Шаг 10 (`c4.md` в пакет); Шаг 12 (Conformance-gate + пункты хендофф-чеклиста).
- `component-tests`: «Граница со слоем юнитов» (развилки = юниты, D1); «Формула» (режимы отказа = Extensions use case + двусторонняя сверка).
- Роли: `plan-reviewer` (conformance-проверка + STOP); `planner` (подключён `documentation`).

| **Трассировка use case → слайс → компонентный тест** | `skills/lib/program-design` + `skills/lib/component-tests` | прошить связь детерминированно: **1 вертикальный срез = 1 внешний вход = 1 системный use case** (Cockburn). Деривация тестов 1:1: **Main Success Scenario → 1 happy-Gherkin**; **каждый Extension `NNa` → ровно 1 сценарий отказа** (несёт код ошибки + exit/исход); формула **`N_сценариев = 1 + #extensions`**; каждый `Then` → узел графа контрактов. **Gate-сверка** в обе стороны: нет extension без сценария и нет сценария без extension (анти-gaming: запрет удалять сценарий без удаления extension). Референс — `pinout-openapi`: UC-1 (`c4.md`) ↔ `validate.feature` (7 = 1+6) ↔ маппинг в `slices/01` |
| **Единый запрос + развилки = логика (урок дефекта D1)** | `skills/lib/program-design` (Шаги 3, 6, 8.1) + уточнение в `skills/lib/component-tests` («Граница со слоем юнитов») | **(1) Жёсткое правило единого запроса (1 модуль = 1 вход на границе программы):** все внешние параметры собираются в **ровно один `Request`**, и только он входит в ингресс. Тулинг — из аргументов/флагов формируем `Request` и передаём в ingress; флаг = **поле `Request`**, не аргумент модуля и не side-инъекция; `os.Args` парсится только в `adapter.go`. **Запрет** проносить внешний выбор мимо `Request` (нет `stdout io.Writer` сбоку — «куда писать» = `req.Out`). Симптом нарушения = D1 (опция уважается одним путём, теряется другим). **(2) Развилка по полю `Request` = чистая логика, юнит по формуле** `1 happy + Σ ветки` (`resolveDestination`, `renderReport`); **не** компонентный сценарий. I/O-труба берёт готовый план и только пишет, без ветвлений по данным. **(3) Запрет test-only второго метода I/O-объекта** (`WriteTo(io.Writer)` рядом с `Write(out)` — стоп; вынеси решение в логику — шов не нужен). **(4) Сверка с формулой component-tests:** матрица `stdout\|файл × json\|md` **не** увеличивает `N = 1 + #extensions` (это юниты); в компонент входит только режим **отказа** записи. Усиливает строку «Трассировка UC→слайс→тест», не конфликтует. Референс дефекта — `rra-docs-another` D1: `egress` (`cli.go:191`) зовёт `WriteTo(…, stdout)`, игнорируя `req.Out`; `WriteTo` заведён «для тестов» и стал боевым путём |
| **Conformance-gate: STOP-проверка дизайна на соответствие скиллу** | `skills/lib/program-design` (новый шаг-гейт перед хендоффом) + `skills/roles/plan-reviewer` | **(1)** Сквозная инфраструктура (общий egress, общие поля `Request`/флаги, `infrastructure.md`) **обязана иметь свою design-карту и контракт** и пройти сверку — не «падать между слайсами». **STOP plan-reviewer:** кросс-срезовый узел без карты/контракта → STOP, возврат в PLANNING (прямой корень D1: у общего egress не было карты → ни одна conformance-проверка на него не запускалась). **(2)** Consolidated **conformance STOP-checklist** в конце `program-design`: механически прогнать **все** жёсткие правила скилла (единый `Request`, один data-аргумент, изоляция I/O, запрет test-only метода, развилка=логика, Gherkin-mapping, граф контрактов). Любое нарушение → **STOP**, возврат к нужному шагу; хендофф запрещён. **(3)** Асимметрия: соответствие проверяет plan-reviewer против `program-design` как эталона, не только самозаполнение Шага 12. **Anti-gaming:** нельзя `[x]` без артефакта. Связь: строка «Единый запрос + развилки» даёт *правила*, этот гейт — их *обязательное применение через STOP* |

## Зависимости

- `release-health-analysis` даёт вердикт, по которому `canary-release` расширяет долю.
- `canary-release` — предусловие наблюдения (артефакт сначала надо выкатить).
- `contract-tests` опирается на `api-specification/` (API-first) и внешний валидатор `pinout`.
- Трассировка UC→слайс→тест: `program-design` выдаёт срез+use case → `component-tests` потребляет его extensions как режимы отказа (антецедент генерации). Цепочка: use case (Cockburn) → срез → Gherkin → узлы графа контрактов.
- «Единый запрос + развилки = логика» (урок D1) — рядом с трассировкой UC→слайс→тест: фиксирует, что развилки по полям `Request` (`--out`/`--format`) уходят в юниты и **не** меняют формулу `N = 1 + #extensions`; делать одним заходом с трассировкой и моделью ошибок (`req.Out`→exit-маппинг).
- «Conformance-gate» форсит правила из «Единый запрос + развилки» через STOP: первая строка даёт *правила*, gate — их *обязательное применение*; gate бесполезен без правил, правила без gate можно обойти. Делать после/вместе с ними.
- `security-ci` + правила безопасной разработки (P1) — две стороны: статика в CI и дисциплина на дизайне/в коде.
- P3 (`load-capacity`/`high-availability`) — вне базового потока; заводятся по факту нагрузки.
