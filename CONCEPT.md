# rationaldev — концепция AI-first рационального SDLC

Дерзкая разработка ПО с ИИ на базе агентов. Команда в своей основе —
**человек + машина**: инженер творчески прорабатывает идеи **с ИИ** и воплощает их
**с ИИ**. Это рациональная автоматизация в духе Никлауса Вирта — не «инструмент рядом
с инженером», а сама ткань работы.

Цель — база для **агентов проектирования и разработки** (набор называется **`izi`**),
названных в честь великих инженеров структурного программирования.

## Научный фундамент

Подход к проектированию и разработке опирается на труды великих инженеров — на наш
взгляд, это фундамент, и другого нет:

| Труд | Авторы |
|---|---|
| *Systematic Programming: An Introduction* | **Niklaus Wirth** |
| *A Structured Approach to Programming* | **Joan K. Hughes**, **Jay I. Michtom** |
| *Structured Programming: Theory and Practice* | **Richard C. Linger**, **Harlan D. Mills**, **Bernard I. Witt** (IBM) |

Плюс **цикл статей автора** по рациональной разработке — именно с него началась эта
дисциплина (ссылки ниже). Ключевые идеи, на которых стоит весь процесс:

- **программа = дерево модулей** (Вирт): декомпозиция сверху вниз, пошаговое уточнение;
- **корректность по построению** (Mills, Cleanroom): доказательная разработка, а не «отладка тестами»;
- **бизнес-логика отделена от I/O**: структурная корректность как инвариант;
- **математическая композиция модулей**: предусловие/постусловие, один data-аргумент, `Result<T,E>`;
- **документация = продукт** для четырёх JTBD-потребителей.

«Качество достигается проектированием, а не тестированием» — поэтому из тестов мы
оставляем **только компонентные** (спецификация чёрного ящика) и **контрактные**, а
корректность модулей доказывается композицией.

## Команда: человек + машина

| Стадия | Кто | Что |
|---|---|---|
| **Discovery** | инженер-разработчик **+ ИИ** | гипотезы, требования, продуктовая проработка (аналитика здесь, отдельной роли-аналитика нет) |
| **Delivery** | агенты `izi` + дирижёр (человек) | реализация задачи от плана до здоровой фичи в проде |

Человек держит решения на границах (намерение, акцепт плана, мерж, приёмка релиза);
рутину Delivery ведут агенты. Асимметрия генератор/критик: кто создаёт артефакт — тот
его не принимает.

## Дерзкая разработка с ИИ: какой это SDLC

Только экспертиза и рациональный научный подход.

- **Нет тестовых сред и стендов.** Только **CI** и **канареечные релизы** прямо в прод за фиче-тогглом.
- **Канареечный выкат за фиче-тогглом** (доля трафика) на **вариативные среды** — VM, контейнер, serverless; без отдельных стендов и без обязательного Kubernetes.
- **Тесты:** только **компонентные** (Gherkin, чёрный ящик) + **контрактные** между компонентами.
- **Контрактные тесты проверяет проект [`pinout`](https://github.com/codemonstersteam/pinout)** и его компоненты (`pinout-openapi`, `pinout-asyncapi`, `pinout-netlist`) — контракты как netlist связей между сервисами.
- **Мониторинг — 4 золотых сигнала**: latency, traffic, errors, saturation. Ничего сверх.
- **Безопасность** — требование на этапе планирования + отдельный **security-scan в CI**; правила безопасной разработки вшиты в скиллы проектирования и реализации.
- **CI использует утилиты экосистемы [RRA](https://github.com/codemonstersteam/rra)** (`rra-docs-another`, `rra-audit-repo`, `rra-docs`) для валидации решений — репозиторий сам проверяется на рациональную дисциплину.

## Агенты `izi` — роли и имена

Общие названия ролей понятны большинству; кодовые имена агентов подобраны по вкладу
инженера в информатику/кибернетику.

| Роль (общая) | Агент | Вклад → почему эта роль |
|---|---|---|
| `planner` | **Wirth** | проектирование: дерево модулей, пошаговое уточнение, минимализм → генерация плана |
| `plan-reviewer` | **Mills** | Cleanroom, корректность «по построению», разработка без отладки → верификация плана |
| `implementer` | **Hughes** | прикладное структурное кодирование → пишет код по плану |
| `fixer` | **Linger** | функционально-теоретическая верификация кода + безопасность (CERT) → ревью кода, классификация дефектов CI |
| `release-health` | **Michtom** | системность и обратная связь → канареечный контур: выкат + 4 золотых сигнала + откат |
| `orchestrator` (человек) | **Witt** (honorary) | со-автор *Theory **and** Practice* → держит границу теория↔практика на Human Gates |

Связь с кибернетикой: **Mills** — математизация корректности; **Michtom** — управление
релизом по обратной связи (ядро кибернетики Винера); **Wirth** — структурная декомпозиция
систем.

**Почему `izi`:** строгая рациональность, поданная легко (`make it as simple as
possible`, по Вирту). Тяжёлый научный фундамент — простой опыт.

## Поток (happy path, 3 человеческих гейта)

```
[Задача] → Входной gate ──► PLANNING(Wirth) ──► PLAN REVIEW(Mills) ──► Human Gate #1 (акцепт плана)
                                                                              │
   IMPLEMENTATION(Hughes) ──► CI ──► CODE REVIEW(Linger) ──► Human Gate #2 (мерж)
                              │  (RRA-утилиты + security-scan)        │
                              └─ классификация ошибки                 ▼
                                              КАНАРЕЙКА в прод за тогглом (Michtom)
                                              выкат ⇄ 4 золотых сигнала: GREEN→расширить / RED→откат
                                                                       │
                                                                       ▼
                                                       Human Gate #3 (приёмка прод-релиза)
```

## Цель

На базе этих ролей и скиллов строятся **агенты проектирования и разработки** в честь
Вирта (и его Оберона), Хьюза, Михтома, Лингера, Миллса и Витта — набор `izi`.
